ACP 深度解读：AI编程Agent的「万能插头」是怎么炼成的

Agent Client Protocol（ACP）正在成为 AI 编程领域的 LSP。Zed 发起、JetBrains 加入、Google/Anthropic/OpenAI 的 Agent 纷纷适配——这个协议凭什么？

一、从一个真实痛点说起

2025年初，一个典型的 AI 辅助编程工作流长这样：

左屏放编辑器写代码，右屏开一个 AI Agent（比如 Claude Code 或 Gemini CLI）对话，浏览器看效果。三个窗口来回切换，上下文不断打断。

更要命的是 绑定问题：

  • 想用 Cursor？那你的 Agent 就是 Cursor 内置的，不能换
  • 想用 Claude Code？得开终端跑 CLI，编辑器和 Agent 之间没有结构化通信
  • 想在 JetBrains 里用 Goose？对不起，没有插件

每个编辑器都要为每个 Agent 写一套插件，每个 Agent 都要为每个编辑器做适配。M × N 的组合爆炸，和 LSP 出现之前的编程语言支持一模一样。

二、LSP 的启示：一个协议解决 M × N 问题

2016 年，微软推出了 Language Server Protocol（LSP）。在此之前，每个编辑器要为每种编程语言单独写语法高亮、补全、跳转定义的插件。有了 LSP，语言工具只需要实现一次协议，就能在所有支持 LSP 的编辑器里使用。

从 M × N → M + N。

ACP 的灵感完全来自这里。它要解决的问题是：让任何 AI 编程 Agent 能在任何编辑器里工作。

不是 MCP（Model Context Protocol）——MCP 解决的是 Agent 怎么访问外部工具和数据（Agent ↔ 工具）。ACP 解决的是 Agent 怎么跟编辑器交互（Agent ↔ 编辑器 UI）。

这个区别至关重要。

三、ACP 的诞生：从 Zed 的内部需求到开放标准

3.1 起源故事

2025 年初，Zed 编辑器团队正在做「agentic editing」实验——让 AI Agent 能自主执行多步代码修改。恰好 Google 找上门来，想让 Gemini CLI 深度集成进 Zed。

问题来了：Gemini CLI 已经能在 Zed 的内嵌终端里跑，但那只是 ANSI 转义码的文本交互，没有结构化的通信。Agent 想展示一个 diff？只能打印文本。Agent 想请求文件访问权限？没有标准方式。

Zed 的开发者回忆说：

“我们需要一种比 ANSI 转义码更结构化的通信方式。”

于是他们定义了一组最小化的 JSON-RPC 端点，用来在 Agent 和编辑器之间传递请求和响应。这就是 ACP 的原型。

3.2 关键决策

Zed 做了一个大胆的决定：把自己内置的 AI 助手也重构为 ACP 协议。

这意味着 Zed 自家的 Agent 和第三方 Agent 走完全相同的接口。没有「特权通道」，没有「内部 API」。这个决定释放了强烈的信号：ACP 不是「给外人用的二等公民接口」，而是 Zed 自己 all-in 的架构选择。

3.3 时间线

时间 里程碑
2025年8月 Zed 发布「Bring Your Own Agent」功能，ACP 作为开放标准公布（Apache License）
2025年8月 Google Gemini CLI 成为首个参考实现
2025年9月 Neovim 社区通过 CodeCompanion 插件接入 ACP
2025年10月 JetBrains 宣布与 Zed 合作，在 IntelliJ/PyCharm 等 IDE 中支持 ACP
2025年10月 Block（Square 母公司）的 Goose Agent 接入 ACP
2025年12月 JetBrains 发文阐述「Agents, Protocols, and Why We’re Not Playing Favorites」
2026年1月 JetBrains 上线 ACP Agent Registry（Beta）
2026年1月 GitHub Copilot CLI 公开预览 ACP 支持

不到半年时间，从一个编辑器的内部实验变成了整个行业的标准。

四、ACP 技术架构：深入协议设计

4.1 通信层：JSON-RPC over stdio

ACP 使用 JSON-RPC 2.0 协议，通信方式是 stdin/stdout 管道。当用户在编辑器里激活一个 Agent 时：

  1. 编辑器以子进程方式启动 Agent
  2. 所有通信通过 stdin/stdout 的 JSON-RPC 消息完成
  3. 支持双向通信——Agent 也可以向编辑器发请求

和 LSP 一模一样的通信模型，成熟、简单、可靠。

4.2 消息类型

ACP 定义了两种消息：

  • Methods（方法调用）：请求-响应模式，期望对方返回结果或错误
  • Notifications（通知）：单向消息，不期望响应。Agent 用它来流式推送更新

4.3 核心会话生命周期

4.4 Agent 端核心方法

方法 说明
initialize 协商版本，交换能力声明
authenticate Agent 认证（如 API Key）
session/new 创建新会话
session/prompt 接收用户输入
session/load 恢复历史会话（可选）
session/set_mode 切换工作模式（如 plan/code/review）

4.5 Client 端核心方法

编辑器作为 Client，需要实现这些方法供 Agent 调用：

方法 说明
session/request_permission Agent 请求执行敏感操作前获取用户授权
fs/read_text_file 读取文件内容
fs/write_text_file 写入文件内容
terminal/create 创建终端执行命令
terminal/output 获取终端输出
terminal/kill 终止终端进程

这个设计非常巧妙——Agent 不直接操作文件系统，而是通过编辑器代理访问。编辑器作为信任中介，可以：

  • 在 Agent 写文件前展示 diff 让用户确认
  • 限制 Agent 只能访问项目目录
  • 拦截危险的终端命令

4.6 流式更新与实时 UI

ACP 大量使用 JSON-RPC 通知来实现实时 UI 更新。Agent 的 session/update 通知可以推送：

  • 文本块（思考过程、代码解释）
  • 代码 diff（多文件修改的结构化表示）
  • 工具调用状态（正在执行什么、执行结果）
  • 执行计划（Agent 打算做哪些步骤）
  • 模式切换（从分析模式切到编码模式）

编辑器收到这些通知后，可以用原生 UI 渲染——不是把 Markdown 文本糊到一个对话框里，而是真正集成到编辑器的 diff 视图、文件树、终端面板中。

五、ACP vs MCP：互补，不是竞争

这是最容易搞混的部分。很多人第一次听说 ACP 会问：「MCP 不是已经有了吗？为什么还需要 ACP？」

答案是：它们解决完全不同的问题。

维度 MCP（Model Context Protocol） ACP（Agent Client Protocol）
解决的问题 Agent 如何访问外部工具和数据 Agent 如何在编辑器中展示 UI
通信方向 Agent ↔ 工具/数据源 Agent ↔ 编辑器/用户界面
类比 给 Agent 装「手」 给 Agent 装「嘴」和「眼睛」
创建者 Anthropic Zed Industries
协议 JSON-RPC JSON-RPC
传输 stdio / HTTP SSE / Streamable HTTP stdio

它们不仅不冲突，还被设计为协同工作：

  1. 编辑器配置了 MCP 服务器（比如数据库查询、API 文档搜索）
  2. 当 ACP 会话建立时，编辑器把 MCP 服务器的配置传给 Agent
  3. Agent 直接连接 MCP 服务器获取工具能力
  4. Agent 通过 ACP 把执行结果展示给用户

ACP 协议文档明确说：「ACP 有意复用 MCP 的数据类型，这样集成者不需要为常见数据类型重新发明轮子。」

⚠️ 别跟 IBM 的 ACP 搞混

另一个值得注意的点：IBM 在 2025 年提出过一个也叫 “ACP” 的协议——Agent Communication Protocol。那个 ACP 是做 Agent 之间通信的（类似 Google 的 A2A），和 Zed 的 Agent Client Protocol 完全不同。

IBM 的 Agent Communication Protocol 已于 2025 年 8 月合并入 Google 主导的 A2A（Agent-to-Agent Protocol），不再独立发展。

所以现在说 ACP，默认就是指 Zed 的 Agent Client Protocol。

六、生态系统：谁在用 ACP？

6.1 Agent 端（已接入 ACP 的 AI 编程助手）

截至 2026 年 3 月，已有 30+ 个 Agent 支持 ACP，包括：

头部玩家：

  • Claude Agent（Anthropic）——通过 Zed 的 SDK 适配器
  • Codex CLI（OpenAI）——通过 Zed 的适配器
  • Gemini CLI（Google）——首个参考实现
  • GitHub Copilot——2026年1月公开预览 ACP
  • Junie（JetBrains）
  • Goose（Block/Square）

中国团队：

  • Kimi CLI（Moonshot AI / 月之暗面）
  • Qwen Code（通义千问）
  • OpenClaw

其他值得关注：

  • Augment Code、Cline、Docker cagent、Mistral Vibe、OpenHands…

6.2 Client 端（支持 ACP 的编辑器/IDE）

编辑器/IDE：

  • Zed——ACP 的发源地
  • JetBrains 全家桶（IntelliJ、PyCharm、WebStorm 等）
  • Neovim——通过 CodeCompanion / avante.nvim / agentic.nvim 插件
  • VS Code——通过 ACP Client 扩展
  • Emacs——通过 agent-shell.el

非编辑器 Client（ACP 的延伸）：

  • Obsidian（笔记应用）
  • marimo（Python notebook）
  • OpenClaw（同时是 Agent 和 Client）
  • Telegram ACP Bot
  • Chrome ACP（浏览器扩展）
  • DuckDB（通过 ACP 扩展查询 Agent）

这个名单说明 ACP 已经突破了「编辑器协议」的边界——任何需要嵌入 AI Agent 交互的应用都可以当 ACP Client。

七、ACP 进阶特性

7.1 会话管理

ACP 支持完整的会话生命周期：

  • session/new：创建新会话
  • session/load：恢复历史会话（需要 Agent 声明 loadSession 能力）
  • session/fork（提案中）：从某个节点分叉会话
  • session/stop（提案中）：停止活跃会话
  • session/delete（提案中）：删除会话

这意味着你可以在 Zed 里开始一个 Claude Code 会话，关闭编辑器，第二天重新打开继续。

7.2 Session Modes（工作模式）

Agent 可以声明多种工作模式，比如：

  • Plan 模式：只分析和规划，不修改代码
  • Code 模式：直接编写和修改代码
  • Review 模式：审查代码，提出建议

编辑器 UI 可以展示模式切换按钮，用户一键切换 Agent 的行为模式。

7.3 Slash Commands

Agent 可以向编辑器注册自定义斜杠命令（类似 Discord 的 /命令）：

  • /test 运行测试
  • /refactor 重构选中代码
  • /explain 解释代码逻辑

编辑器在 prompt 输入框里提供自动补全和提示。

7.4 Agent Plan（执行计划）

Agent 可以在执行前先告诉编辑器「我打算做什么」：

  1. 读取 src/auth.ts 了解当前认证逻辑
  2. 修改 src/auth.ts 添加 JWT 验证
  3. 创建 src/auth.test.ts 编写单元测试
  4. 运行 npm test 验证

编辑器以清晰的 UI 展示这个计划，用户可以审查、修改或取消。

7.5 MCP-over-ACP（提案中）

一个正在讨论的 RFD（Request for Dialog）提案：在 ACP 通道上转发 MCP 消息。这样 Agent 不需要单独建立 MCP 连接，一根管道搞定所有通信。

7.6 可扩展性

  • 用 _meta 字段携带自定义数据
  • 用 _ 前缀定义自定义方法
  • 在 initialize 时声明自定义能力

八、对比 A2A：不同层次的标准化

Google 的 A2A（Agent-to-Agent Protocol）解决的是 Agent 之间的通信——比如一个「旅行规划 Agent」怎么调用一个「航班搜索 Agent」。

把三个协议放在一起，层次就清晰了：

协议 连接 类比
ACP Agent ↔ 编辑器 Agent 的「嘴巴和眼睛」
MCP Agent ↔ 工具 Agent 的「手」
A2A Agent ↔ Agent Agent 的「社交能力」

三层协议，各司其职，合在一起构成 AI Agent 的完整通信栈。

九、实际体验：在 Zed 里用 Claude Code

以最典型的场景为例，ACP 的实际体验是这样的：

  1. 安装 Claude Code：npm install -g @anthropic-ai/claude-code
  2. 在 Zed 配置：添加 Claude Code 为外部 Agent
  3. 打开项目，在 Agent 面板发送 prompt
  4. Claude Code 通过 ACP 接收 prompt → 分析项目 → 通过 session/update 流式推送分析结果 → 通过 fs/read_text_file 读取相关文件 → 生成修改 → 通过 session/request_permission 请求写入权限 → 编辑器展示 diff → 用户确认 → 写入

全程在编辑器里完成。没有窗口切换，没有复制粘贴，Agent 的每一步操作都在编辑器的原生 UI 中可视化。

十、ACP 的意义与未来

10.1 为什么 ACP 能成功？

  1. 解决了真实痛点——Agent 和编辑器的集成确实是 M × N 问题
  2. Zed 自己 all-in——不是做给别人用的二等接口
  3. JetBrains 的背书——全球最大的商业 IDE 厂商加入
  4. 时机恰好——AI 编程 Agent 在 2025 年爆发，行业急需标准
  5. 设计克制——基于 JSON-RPC，不过度设计，不搞复杂的抽象

10.2 挑战

  • Web IDE 支持——ACP 目前基于 stdio，浏览器环境（如 GitHub Codespaces）需要额外适配
  • 非编码场景——ACP 高度优化了编程工作流，能否扩展到其他领域？
  • 与 A2A 的边界——当一个 Agent 需要同时和编辑器交互（ACP）又和其他 Agent 协作（A2A），集成方式还需要探索

10.3 我的判断

ACP 是 2025-2026 年 AI 基础设施领域最重要的协议之一。

它不像 MCP 那样声势浩大（毕竟 Anthropic 有足够的 PR 火力），也不像 A2A 那样有 Google 的号召力。但 ACP 解决的问题实在、方案务实、生态增长迅猛。

如果 LSP 是编程语言的「万能插头」，ACP 就是 AI 编程 Agent 的「万能插头」。

从 2025 年 8 月发布至今不到 7 个月，30+ Agent、15+ Client 接入。按这个势头，ACP 会成为 AI 编程领域的事实标准——就像 LSP 在语言工具领域做到的那样。
