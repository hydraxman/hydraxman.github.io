AI Agent 通信协议大战：ACP vs A2A，从分裂到统一的故事

2025 年是 AI Agent 的元年，但有一个尴尬的事实：每个 Agent 都是一座孤岛。

LangChain 造的 Agent 不认识 CrewAI 的 Agent，AutoGen 的多智能体系统和自定义框架之间毫无沟通。企业里跑着几十上百个 Agent，想让它们协作？对不起，每对 Agent 之间都要写一套定制集成。n 个 Agent 就可能需要 n(n-1)/2 个连接点。

这不是技术问题，是标准缺失的问题。

于是在 2025 年，两大科技巨头几乎同时给出了自己的答案：IBM 推出了 ACP（Agent Communication Protocol），Google 发布了 A2A（Agent-to-Agent Protocol）。两个协议的目标惊人地一致——打破 Agent 孤岛，让不同框架的智能体自由对话。但它们的设计哲学和技术路线截然不同。

更戏剧性的是，这个”双雄争霸”的故事只持续了五个月，就以合并统一收场。

今天这篇文章，我们来完整复盘这场 Agent 协议之战。

一、ACP：IBM 的”简单即正义”路线

起源与背景

2025 年 3 月，IBM Research 推出了 BeeAI 平台——一个开源的 Agent 编排系统。为了解决平台内部和跨框架 Agent 的通信问题，他们设计了一套通信协议，就是 ACP（Agent Communication Protocol）。

ACP 的设计理念用一句话概括：让 Agent 通信像调 REST API 一样简单。

核心技术特点

1. 纯 REST 架构

ACP 完全基于标准 HTTP/REST 接口。不需要 JSON-RPC，不需要 WebSocket，不需要任何复杂的通信机制。用 cURL 就能和 Agent 交互：

这意味着任何会写 HTTP 请求的开发者，五分钟内就能和 ACP Agent 对话。不需要学新协议，不需要装 SDK（虽然 Python 和 TypeScript SDK 都有提供）。

2. 客户端-服务器模型

ACP 采用经典的 Client-Server 架构：

  • ACP Client：Agent、应用或服务，发起请求
  • ACP Server：托管一个或多个 Agent，执行请求并返回结果

这种模型简单直观，和传统微服务架构天然兼容。

3. 离线发现（Offline Discovery）

这是 ACP 最有创意的特性之一。Agent 可以将自己的元数据（能力描述、API Schema 等）嵌入到分发包中，通过 YAML 文件发布在已知 URI 上。即使 Agent 处于离线或休眠状态，其他系统也能发现它的存在和能力。

这对按需扩缩（Scale-to-Zero）场景特别友好——Agent 不用一直跑着等被发现，系统找到它之后再按需唤醒。

4. 异步优先，同步兼容

ACP 默认使用异步通信，适合长时间运行的复杂任务。Agent 可以暂停执行、等待额外数据，支持 Human-in-the-Loop（人在回路）场景。同时也完整支持同步请求。

5. 原生多模态

ACP 使用 MIME Type 标识内容类型，天然支持文本、图片、音频、视频、结构化数据等任意格式的消息交换，不需要协议层面的修改。

ACP 的适用场景

  • 企业内部多 Agent 编排：IT 运维、流程自动化
  • 受监管行业：金融服务、合规工作流（支持离线/气隙环境）
  • 轻量级集成：不想引入复杂依赖的团队
  • Agent 市场/平台：BeeAI 这类 Agent 共享平台

二、A2A：Google 的”互联网级”野心

起源与背景

2025 年 4 月，也就是 ACP 发布仅一个月后，Google 发布了 A2A（Agent-to-Agent Protocol）。与 IBM 的低调内生不同，Google 一出手就带着 50+ 科技合作伙伴（包括 Oracle、MongoDB、Salesforce、KPMG、Deloitte 等），声势浩大。

A2A 的野心更大：不仅仅是让 Agent 通信，而是要建立一套互联网级别的 Agent 互操作标准。

核心技术特点

1. JSON-RPC + HTTPS

A2A 采用 JSON-RPC 作为消息信封，通过 HTTPS 传输。相比 ACP 的纯 REST，JSON-RPC 提供了更丰富的语义：方法调用、参数传递、错误处理都有标准化的格式。

2. 对等（Peer-to-Peer）通信

与 ACP 的 Client-Server 不同，A2A 采用对等架构。任何 Agent 既是客户端又是服务端，可以直接向其他 Agent 发起请求或接收请求，没有中心化瓶颈。

这种设计更适合去中心化的多 Agent 系统，Agent 之间可以直接协商、委托、协作。

3. Agent Card（智能名片）

每个 A2A Agent 都有一个 Agent Card——一个 JSON 文件，发布在 /.well-known/agent.json 路径下，描述 Agent 的：

  • 身份和能力
  • 支持的交互模式
  • 认证要求
  • API 端点

Agent Card 就像 Agent 世界的”数字名片”，让其他 Agent 能快速了解对方的能力边界。

4. 任务生命周期管理

A2A 将 Agent 交互抽象为任务（Task）概念，定义了完整的任务生命周期：

  • 提交：Client Agent 创建任务
  • 处理：Remote Agent 执行
  • 流式更新：通过 SSE（Server-Sent Events）实时推送进度
  • 产出物（Artifact）：任务的输出结果
  • 完成/失败/取消：明确的终态

这套设计让复杂的多步骤任务有了清晰的状态追踪。

5. 动态协商

A2A 支持复杂的交互模式，包括”提议-接受-反提议”的协商流程。Agent 之间不是简单的”请求-响应”，而是可以进行多轮交互来达成共识。

6. 标准化身份

A2A 提供统一的 Agent 寻址和密码学身份验证方案，对跨组织的安全通信至关重要。

A2A 的适用场景

  • 跨组织 Agent 协作：供应链、合作伙伴集成
  • 大规模多 Agent 系统：云端分布式 Agent 网络
  • 复杂协商场景：需要多轮对话达成共识的任务
  • 企业级应用：有 Google 生态和 50+ 合作伙伴背书

三、正面对决：ACP vs A2A 关键差异

维度 ACP（IBM） A2A（Google）
发起方 IBM Research / BeeAI Google + 50+ 合作伙伴
发布时间 2025 年 3 月 2025 年 4 月
通信协议 REST / HTTP JSON-RPC / HTTPS
架构模型 客户端-服务器 对等（P2P）
发现机制 离线发现（YAML 元数据） Agent Card（.well-known/agent.json）
SDK 依赖 不需要（可选） 不需要（可选）
任务模型 Runs（运行实例） Tasks（任务生命周期）
流式支持 SSE SSE
异步支持 异步优先 异步优先
多模态 MIME Type 原生支持 Parts 支持多种类型
状态管理 有状态 + 无状态 基于任务的状态管理
安全模型 标准 HTTP 安全 密码学身份 + OAuth
治理 Linux Foundation (LF AI) Linux Foundation
学习曲线 低（纯 REST） 中等（需理解 JSON-RPC + 任务模型）
适合场景 企业内部、边缘、受监管环境 跨组织、云端、大规模分布式

简单来说：

  • **ACP 是”HTTP 世界的公民”**——设计哲学是把 Agent 通信降维到开发者最熟悉的 REST 模式，降低门槛到最低。
  • **A2A 是”互联网世界的公民”**——设计目标是建立一套能支撑全球 Agent 互联的完整标准，功能更丰富但复杂度也更高。

四、剧情反转：ACP 并入 A2A

故事到这里，本应是一场旷日持久的标准之争。但 2025 年 8 月，剧情急转直下。

IBM 宣布 ACP 正式合并入 A2A，统一在 Linux Foundation 旗下。

2025 年 9 月 1 日起，ACP 团队停止独立开发，将技术成果和专业经验贡献给 A2A 项目。这不是一方吞并另一方，而是一次务实的融合。

为什么合并？

  1. 避免生态分裂：两个协议解决的是同一个问题——Agent 间通信。维持两套竞争标准只会让开发者困惑，让生态碎片化
  2. 互补优势：ACP 的 REST 简洁性和离线发现能力，可以作为 A2A 的补充特性融入统一标准
  3. 治理一致：两者最终都归入了 Linux Foundation，合并是自然结果
  4. 产业共识：50+ 合作伙伴已经站队 A2A，单独维护 ACP 的投入产出比不高

合并后的影响

  • A2A 成为事实标准：Agent 间通信协议的标准之争基本尘埃落定
  • ACP 用户需迁移：IBM 提供了官方迁移指南（从 ACP 到 A2A）
  • BeeAI 平台继续存在：作为 A2A 生态中的一个实现，BeeAI 从 ACP 切换到 A2A
  • ACP 的优秀特性被吸收：REST 友好性、离线发现等理念正在影响 A2A 的演进方向

五、更大的图景：Agent 通信协议全景

ACP 和 A2A 的合并，只是 2025-2026 年 Agent 协议生态整合的一个缩影。目前整个生态的格局是：

四大协议各司其职

协议 主导方 核心定位
MCP Anthropic Agent ↔ 工具/数据源（上下文接入）
A2A（含 ACP） Google + Linux Foundation Agent ↔ Agent（跨框架通信）
ANP 社区驱动 Agent ↔ 互联网（开放网络发现）
AG-UI CopilotKit Agent ↔ 人类 UI（前端交互）

它们不是竞争关系，而是互补的协议栈：

  • MCP 让 Agent 有了”手”——可以操作工具、读取数据
  • A2A 让 Agent 有了”嘴”——可以和其他 Agent 交流协作
  • AG-UI 让 Agent 有了”脸”——可以和人类自然交互
  • ANP 让 Agent 有了”网络”——可以在互联网上被发现和连接

企业该如何选择？

对于 2026 年的企业来说，建议是：

  1. 先接 MCP：让你的 Agent 能用工具、读数据，这是基础
  2. 再接 A2A：需要多 Agent 协作时引入，特别是跨团队、跨框架场景
  3. 关注 AG-UI：如果你的 Agent 需要面向终端用户
  4. 观望 ANP：还在早期，适合前沿探索

七、另一个 ACP：Agent Client Protocol 与命名之惑

写到这里，必须聊一个让很多开发者困惑的事情：ACP 这个缩写，实际上被两个完全不同的协议占用了。

IBM 的 ACP vs Zed 的 ACP

Agent Communication Protocol Agent Client Protocol
缩写 ACP ACP
发起方 IBM Research / BeeAI Zed Industries（Zed 编辑器团队）
定位 Agent ↔ Agent（智能体间通信） Editor ↔ Agent（编辑器与编码 Agent 通信）
类比 像 HTTP——让不同服务互相对话 像 LSP——让不同编辑器接入同一种语言服务
通信方式 REST / HTTP JSON-RPC over stdio / HTTP / WebSocket
现状 已合并入 A2A 活跃发展中，被多个 IDE 采用

是的，你没看错。一个解决 Agent 之间怎么说话，一个解决编辑器怎么和 Agent 说话。 完全不同的层面，却用了同一个名字。

Agent Client Protocol：AI 时代的 LSP

如果你是开发者，一定用过 LSP（Language Server Protocol）。VSCode 之所以能支持几十种编程语言的智能补全，就是因为 LSP 把”语言智能”从编辑器中解耦了出来——任何编辑器 + 任何语言服务器，通过 LSP 就能互通。

Agent Client Protocol 想做完全一样的事情，但针对的是 AI 编码 Agent。

在 Agent Client Protocol 出现之前，情况是这样的：

  • Codex 只能在特定界面里用
  • Claude Code 有自己的 CLI 接口
  • Gemini CLI 又是另一套
  • 每个编辑器想接入新的 AI Agent，都要写一套定制集成

实现了 Agent Client Protocol 的 Agent，可以被任何兼容编辑器直接使用。JetBrains AI Assistant 已经原生支持了这个协议，用户可以在 IDE 内直接选择和切换不同的 ACP Agent。

OpenClaw 的实战：ACP 桥接器

OpenClaw（一个开源的 AI Agent 平台）提供了一个有趣的实践案例。它实现了 Agent Client Protocol 桥接器，让用户可以通过一个统一入口管理多个编码 Agent：

配置完成后，你可以在 Zed 的 Agent 面板中直接和 OpenClaw 对话——而 OpenClaw 背后可以路由到 Codex、Claude Code、Gemini CLI 等不同的编码 Agent，用户完全不用关心底层用的是哪个。

OpenClaw 的 ACP 实现支持：

  • 多 Agent 路由：通过 session key 将请求路由到不同的 Agent（Pi、Claude、Codex、OpenCode、Gemini）
  • 会话持久化：支持 persistent 模式，保持上下文连贯
  • 线程绑定：在 Discord 等平台上，可以将 ACP 会话绑定到特定线程
  • 运行时配置：动态切换模型、调整权限、设置超时

这种”ACP 桥接器”模式的价值在于：编辑器只需要支持一个协议（ACP），就能通过 OpenClaw 接入整个 Agent 生态。 不用为每个 Agent 单独写集成。

两个 ACP 的本质区别

用一个比喻来总结：

  • IBM 的 Agent Communication Protocol（已合并入 A2A）解决的是”Agent 组队打副本”的问题——多个 Agent 如何协作完成复杂任务
  • Zed 的 Agent Client Protocol 解决的是”玩家操作角色”的问题——人类开发者如何通过编辑器驱动 AI Agent 工作

两者分别处于不同的通信层：

当你在 Zed 里对 Codex 说”重构这个模块”，Agent Client Protocol 负责把你的指令传给 Codex。如果 Codex 需要调用一个专门做代码审查的 Agent 来协助，那就是 A2A 的事了。如果它需要读取你的数据库 Schema，那是 MCP 的活。

三个协议各管一段，从人到 Agent 到 Agent 到工具，形成完整的通信链路。

八、更新后的思考

回看整个 Agent 通信协议的全景，几个新的观察：

1. 命名冲突本身就是行业混乱的缩影。 两个完全不同的协议都叫 ACP，说明这个领域发展太快、标准化太慢。好消息是，IBM 的 ACP 已经并入 A2A，至少减少了一层混淆。

2. 协议栈正在从”协议大战”走向”分层治理”。 不再是谁取代谁，而是每个协议找到自己的层：Agent Client Protocol 管人机界面，A2A 管 Agent 间通信，MCP 管工具接入。就像 Web 世界的 HTTP/TCP/IP 分层一样自然。

3. 实际落地比协议设计更重要。 OpenClaw 用 Agent Client Protocol 桥接多个编码 Agent 的做法，比协议规范本身更能说明问题。开发者不关心你用的是 REST 还是 JSON-RPC，他们关心的是：我能不能用一个入口管理所有 AI 编码助手？

Agent 协议的终局不是某个协议”赢了”，而是它们像 TCP/IP 协议栈一样，变成开发者不用思考就在用的基础设施。

到那一天，我们就不会再写”ACP vs A2A”的对比文章了——就像今天没人写”HTTP vs TCP”的对比一样。
