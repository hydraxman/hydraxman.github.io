Superpowers：9万星的AI编码框架，把你的Agent变成高级工程师

今天 GitHub Trending 上有个项目特别扎眼：Superpowers，3天涨了近5000颗星，总量突破94,000。单日最高涨幅1,867星。

这不是又一个 AI Agent 框架。它做的事情更有趣——**给现有的AI编码Agent注入”资深工程师的工作习惯”**。

它解决了什么问题？

用过 Claude Code、Cursor、Codex 的人都知道一个痛点：AI编码Agent像一个热情但莽撞的实习生。

你说”帮我写个用户认证”，它二话不说就开始写代码。写完你一看：没测试、没错误处理、架构随意、变量命名混乱。更要命的是，它写完会自信满满地说”Done! Everything should work perfectly!”——但你一跑，全是bug。

Superpowers 的作者 Jesse Vincent 观察到了这个问题，他的解决方案很巧妙：不是造一个新的Agent，而是给现有Agent注入一套行为规范。

核心理念：三条铁律

铁律一：设计先行

这条最反直觉。一个 Todo 应用、一个工具函数，也需要设计？Superpowers 说：是的，尤其是看起来简单的项目。

因为”简单”项目最容易栽在未审视的假设上。一个 Todo 应用，你是要本地存储还是云端？要不要多用户？要不要离线支持？这些问题不问清楚就开始写，后面全是返工。

Superpowers 的 brainstorming 技能会强制Agent走这个流程：

  1. 探索项目上下文
  2. 逐一问清楚需求（不是一次甩一堆问题）
  3. 提出 2-3 个方案和权衡
  4. 人类确认后才开始编码

有一个叫 <HARD-GATE> 的标记，直接锁死——没过设计关就不许调用任何实现技能。

铁律二：没有失败测试就不写代码

这就是经典的 TDD（Test-Driven Development），但 Superpowers 把它执行到了极端：

  • 写了一个函数但没有先写测试？删掉这个函数，从测试开始重写
  • 不能”保留参考”——先写的代码会污染你对测试的思考
  • 每个任务粒度细到2-5分钟：”写失败测试”是一步，”运行确认失败”是另一步

铁律三：没有证据就不能说”完成了”

要宣称完成？跑命令，贴输出，用证据说话。

这条铁律还附带一个禁止列表——Agent不能在验证前说：

  • ❌ “Great!”
  • ❌ “Perfect!”
  • ❌ “Done!”
  • ❌ “Everything looks good!”

这些词只有在跑完测试、确认通过后才允许出现。

架构亮点：子Agent驱动开发

Superpowers 最精彩的设计是 Subagent-Driven Development。

传统做法是让一个Agent从头干到尾。问题是：干到第50个文件时，Agent的上下文已经被前面49个文件的信息填满了，注意力开始涣散，质量急剧下降。

Superpowers 的做法：

每个任务 spawn 一个全新的子Agent，上下文完全隔离。主Agent只做协调，不做实现——它精心构造每个子Agent需要的上下文，而不是把自己的Session历史甩过去。

更妙的是两阶段Review：

  1. Spec 合规检查（另一个子Agent）：做的东西对不对？
  2. 代码质量检查（又一个子Agent）：做的东西好不好？

先确认”做对了”，再追求”做好了”。这个顺序很重要——代码写得再漂亮，如果做的不是要求的东西，就是废品。

实际测试数据：7个子Agent完成一个实现计划，总Token消耗约150万，成本仅 $4.67。

反恭维设计：一个被忽视的创新

Superpowers 有一个不起眼但极其重要的技能：Receiving Code Review（接收代码审查）。

它明确禁止Agent在收到Review反馈时说：

  • “You’re absolutely right!”
  • “Great point!”
  • “Excellent feedback!”
  • “Let me implement that now”（在验证前）

取而代之的流程是：

  1. 读完反馈，不要反应
  2. 用自己的话重述要求
  3. 验证：这个建议在当前代码库里是否合理
  4. 技术评估：是正确的吗？
  5. 如果不对——推回去，给出理由

这打破了AI Agent最常见的恶习：无脑同意一切。一个好的工程师会说”你这个建议在这个场景下不适用，因为…”，而不是对着每个Review意见说”Great point!”然后盲目实现。

技术实现：优雅的简洁

整个项目的实现出人意料地简单：

  • 14个 Markdown 文件（SKILL.md），每个定义一个”技能”
  • 一个 session-start hook（bash脚本），在Agent启动时注入技能系统
  • 支持 Claude Code、Cursor、Codex、OpenCode、Gemini CLI

没有复杂的orchestration引擎，没有自定义Runtime，没有数据库。就是一组写得极其精准的指令文档，通过现有Agent的插件系统注入。

这也揭示了一个深刻的洞察：约束AI Agent行为最有效的方式，不是代码级的强制执行，而是精心设计的Prompt级规范。 因为Agent的行为本质上是由指令驱动的，好的指令 = 好的行为。

与其他框架的对比

特性 Superpowers LangChain AutoGPT
定位 行为规范注入 工具链编排 自主任务执行
复杂度 14个Markdown文件 数万行代码 数千行代码
依赖 无（纯文档） Python生态 Python + 多API
子Agent 精心构造上下文 链式调用 自我规划
质量控制 强制TDD+两阶段Review 无内置 无内置

Superpowers 不和它们竞争——它可以跑在 Claude Code 上面的同时，项目里照样用 LangChain。它解决的是Agent怎么工作的问题，不是Agent用什么工具的问题。

为什么它火了？

时机。2026年3月，AI编码Agent从”尝鲜”进入”日常使用”阶段。越来越多的开发者发现：Agent能写代码不等于能写好代码。 就像给实习生一台电脑不等于他能交付生产级代码。

Superpowers 提供了这个缺失的环节——不是更强的模型，不是更多的工具，而是更好的工作习惯。

这也许是AI编码领域下一个重要方向：不是让Agent更聪明，而是让Agent更有纪律。
