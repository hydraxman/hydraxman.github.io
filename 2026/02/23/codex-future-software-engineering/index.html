当Codex成为"队友"：OpenAI内部正在发生的软件工程革命

本文基于 The Pragmatic Summit（2026年2月11日，旧金山）上 OpenAI 应用 CTO Vijaye Raji 和 Codex 工程负责人 Thibault Sottiaux 的对谈，从软件工程从业者的角度重新解读。

当Codex成为”队友”：OpenAI内部正在发生的软件工程革命

2026年2月11日，旧金山，500名工程领导者聚在一起，听了一场让人坐不住的对话。

OpenAI的应用CTO和Codex团队负责人，当着所有人的面，把他们内部的工作方式扒了个底朝天。

结论是：我们以为AI编程还在”自动补全”阶段。OpenAI内部已经进入了”委托制”。

合上笔记本，回来活干完了

VJ（Vijaye Raji，OpenAI应用CTO）描述了过去6个月他亲眼见证的变化：

工具 → 功能扩展 → Agent → 队友

这不是PPT上的路线图，而是已经发生的事。

他们内部有个新功能叫 Codex Boxes——在服务器端预留开发环境，工程师在本地编排好任务，然后合上笔记本去开会。回来时，多个任务已经并行完成。

“People shut down their laptop, go to a meeting, come back and then all of the work has been done.”

VJ的原话。不是愿景，是日常。

更夸张的数字：OpenAI内部有使用排行榜，顶级工程师每周消耗的token达到数千亿级别。注意，是”亿”。

瓶颈永远在转移，这才是关键

这场对话里最有洞察力的观点，不是”AI多厉害”，而是Tibo（Thibault Sottiaux，Codex工程负责人，33人团队）关于瓶颈转移的描述。

他说团队几乎每周都在重新发明自己的工作方式。

为什么？因为：

每一层效率提升都会暴露下一层的低效。

这其实是一个深刻的系统性观察——很多团队引入AI编程工具后觉得”没那么神”，可能不是工具的问题，而是你只解决了第一层瓶颈，后面的还没碰到。

对软件工程的启示：AI编程不是一个点优化，而是一个系统性重构。 只用Copilot补全代码，相当于给了你一辆跑车但路还是土路——你需要同时升级整条路。

面试者开始问”你给我多少算力”

Tibo讲了一个让他自己都愣住的面试场景：

一个候选人问：**”在OpenAI做产品，你们能给我多少算力？”**

过去这种问题只有训练大模型的研究员才会问。现在产品工程师也开始关心人均算力配额了。

这说明什么？

算力正在成为个人生产力的核心变量。

就像10年前”公司给不给配MacBook”是一个吸引力因素，现在变成了”公司给不给足够的AI推理资源”。

VJ补充了一个判断：如果你有好品味、懂产品、会表达，现在的杠杆率是前所未有的。不是因为AI替代了你，而是AI放大了你。

设计师写的代码比6个月前的工程师还多

这大概是整场对话最”炸裂”的一句话。

OpenAI内部的设计师，现在用Codex直接把设计稿变成可运行的原型。不是等工程师排期，不是画完图扔过墙。设计师自己写，自己验证，自己迭代。

而一个叫Alexander Embiricos的PM，是Codex团队唯一的产品经理——一个人管33个工程师。

他怎么做到的？

  1. Bug bash活动结束 → Codex汇总反馈到Notion
  2. Codex自动拆分成bug报告和功能改进
  3. 录入Linear，分配给对应工程师
  4. 后续用Codex跟进每个人的进度

“He’s becoming like a 10x, like 50x program manager just by leveraging AI.”

一个PM，50倍效率。

夜间，Codex在自己跑测试

Tibo描述了一个已经成为常规的做法：

Codex在夜间自主运行QA测试循环，持续测试、标记回归问题。工程师早上来看结果就行。

但更让人深思的是一位研究员的感受：

“Every time I think I’m more capable than Codex, I figure out I’m wrong and I just didn’t prompt it right.”

每次我以为自己比Codex强，最后都发现是我错了，只是提示词没写对。

这位研究员发现Codex已经能独立训练模型，然后写一份PDF报告，包含发现和洞察。团队做的事变成了：看报告 → 找方向 → 给Codex新任务 → 继续循环。

AI在改进AI。 人类的角色从”执行者”变成了”审核者”和”方向设定者”。

这对普通工程师意味着什么？

OpenAI有无限Token——这是他们的特权。但对话中暴露的趋势，和Token多少无关：

1. “会写代码”正在贬值，”会用代码”正在升值

VJ在行业干了25年，经历过IntelliSense、高级语言、框架抽象。每一次有人说”用这个的不是真正的开发者”，最后都被证明是错的。

基础永远重要——但”基础”的定义在变。数据结构和算法依然重要，但产品直觉、系统思维、在抽象层之间灵活移动的能力正在变得更稀缺、更值钱。

2. 并行探索取代了设计文档争论

以前遇到技术选型，团队写设计文档，开会讨论两小时，排除备选方案。

现在OpenAI的做法：让Codex同时实现多个方案，然后看哪个效果好。

决策从”讨论后择一”变成了”实现后比较”。

这个变化的深意是：试错成本被AI大幅压低后，”先做再想”可能比”先想再做”更高效。

3. 管理结构必须跟着变

Tibo的团队33人直接汇报给他一个人。为什么？

当个体的生产力因AI大幅提升时，传统的层级管理结构很容易成为瓶颈。

一个中层经理卡住所有决策审批？在这个速度下行不通了。扁平化不是管理理念，是效率倒逼。

4. 新人可能比老人适应得更快

OpenAI今年夏天要招约100名应届生。Tibo提到一个叫Ahmed的新人，6个月前加入，表现惊人。

“My brain is probably already in decline… this person Ahmed’s brain is just absolute peak.”

新人的优势：没有旧习惯需要覆盖，天然就是”AI原生”。他们入职的第一个工具就是Codex——用它问问题、浏览代码库、了解同事在做什么。

6个月后的预测

Tibo说两年太远了，他只敢预测6个月：

  1. 速度再提升一个数量级
  2. 多Agent协作网络——大量Agent协同完成超大目标。比如”24小时从零重建一个浏览器”，产出数百万行代码
  3. 代码将被抽象化——你不需要看代码本身，而是通过形式化验证证明它正确，只关注输入和输出

VJ补充了一个担忧：当系统足够复杂时，调试会变得极其困难。未来的工程师可能更像医生诊断病人——靠”症状”来定位问题。

Tibo最后加了一个近期预测：年内就会出现个人助理层。你不需要管理一百个小Agent，而是有一个总控助理，代表所有后台Agent的工作，你只需要和它对话。

三个悬而未决的问题

这场对话没有回避的难题：

1. 无限Token特权能复现吗？

OpenAI内部人人无限Token，这催生的工作方式在成本敏感的现实世界中能走通吗？VJ的建议是换个角度算账：不要问”用了多少Token”，要问”一个24小时工作的队友值多少钱”。

2. 当代码不需要人看时，谁来保证安全？

代码被抽象到人不需要读的程度时，安全性和可审计性怎么办？Tibo认为答案是形式化验证——用数学证明代码正确，而不是靠人眼review。

3. AI原生一代，更强还是更脆弱？

跳过了前辈的基本功训练，长远来看是好事还是坏事？Tibo的回答：基础依然极其重要，但环境设计比个人训练更关键——好的代码架构和护栏，能让新人在框架内发挥出惊人的生产力。

写在最后

VJ在这个行业干了25年，经历过互联网泡沫、Y2K、移动革命、社交网络革命。他对这一次的判断是：

“I don’t think I’ve ever seen anything like this. Some of these charts don’t make sense.”

我觉得我从来没有见过这样的事情。有些增长曲线根本说不通。

软件工程正在从”写代码”变成”编排Agent”。 不是未来，不是愿景，是OpenAI内部已经在发生的现在进行时。

问题不是”AI会不会改变软件工程”。

问题是：你准备好了吗？
