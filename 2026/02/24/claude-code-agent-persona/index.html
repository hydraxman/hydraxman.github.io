你以为 Claude Code 只是一个 AI 编程助手？不，它是一个可以变身的编程助手。架构师、代码审查员、安全专家、DevOps 工程师……只需要一个 Markdown 文件，Claude Code 就能切换人格，变成你需要的任何角色。

一、为什么需要 Agent 人格定义？

用过 Claude Code 的开发者都有这样的体验：让它写代码很强，但你发现它什么都想做。你让它查个 bug，它顺手重构了半个文件；你让它分析架构，它直接开始改代码。

这就好比你请了一个全能员工，但他不知道自己的边界在哪。

Subagent 系统就是 Anthropic 给出的答案——让你为 Claude Code 定义不同的人格角色，每个角色有明确的职责、工具权限和行为约束。

核心价值：

  • 🎯 职责隔离：代码审查员只读不写，不会”顺手”改你的代码
  • 🧠 上下文保护：每个 Agent 有独立的上下文窗口，探索任务不会污染主对话
  • 💰 成本控制：简单任务用 Haiku（便宜快速），复杂任务才用 Opus
  • 🔄 可复用：定义一次，所有项目通用

二、文件结构：一个 Markdown 就是一个人格

Claude Code 的 Agent 定义极其简洁——就是一个带 YAML frontmatter 的 Markdown 文件。

文件存放位置

位置 作用域 优先级 说明
--agents CLI 参数 当前会话 最高 临时测试用
.claude/agents/ 当前项目 高 团队共享，提交到 Git
~/.claude/agents/ 所有项目 中 个人全局配置
插件 agents/ 目录 插件范围 最低 第三方插件提供

优先级规则：同名 Agent，高优先级覆盖低优先级。这意味着你可以在项目级别覆盖全局配置，实现项目定制化。

文件格式

就这么简单。--- 之间是 YAML 配置（元数据），下面是 Markdown 格式的系统提示词（人格定义）。

核心配置字段

字段 必填 说明
name ✅ Agent 名称，用于识别和调用
description ✅ 描述何时应该使用此 Agent（Claude 据此自动委派）
tools ❌ 允许使用的工具列表，如 Read, Write, Edit, Bash, Glob, Grep
disallowedTools ❌ 禁止使用的工具（与 tools 互斥）
model ❌ 指定模型：haiku（快+便宜）、sonnet（均衡）、opus（最强）
permissionMode ❌ 权限模式：default、plan（只读）、auto（自动批准）
maxTurns ❌ 最大对话轮数，防止 Agent 无限循环
mcpServers ❌ 可访问的 MCP 服务器列表
hooks ❌ 生命周期钩子（启动前/后等）

三、三大内置 Agent：开箱即用

Claude Code 自带三个内置 Subagent，不需要任何配置就能自动使用：

1. Explore（探索者）

这是最常被自动触发的 Agent。当 Claude 需要理解代码库结构时，它会委派给 Explore，而不是自己去翻文件。好处是：

  • 探索过程不会占用主对话的上下文窗口
  • 用 Haiku 模型，速度快且省钱
  • 有三个彻底性级别：quick（快速定位）、medium（平衡探索）、very thorough（全面分析）

2. Plan（规划者）

当你使用 /plan 命令进入规划模式时，Plan Agent 负责收集信息但不做任何修改。它的设计有一个巧妙之处：Subagent 不能再生成 Subagent（防止无限嵌套），所以 Plan 模式通过单独的 Agent 来收集上下文。

3. General-purpose（通用型）

当任务需要「既探索又修改」时，Claude 会委派给 General-purpose Agent。它拥有所有工具权限，适合复杂的、有依赖关系的多步骤操作。

六、核心实现原理

自动委派机制

Claude Code 的 Agent 委派不是简单的关键词匹配，而是语义理解。工作原理：

关键设计决策：

  1. description 是路由依据：Claude 读取所有 Agent 的 description 字段，用语义匹配决定委派。所以 description 写得越清晰，自动委派越准确。

  2. Subagent 不能嵌套：为防止无限递归，Subagent 不能再生成 Subagent。这是一个硬性限制。

  3. 上下文完全隔离：每个 Subagent 运行在独立的上下文窗口中，只接收自己的系统提示词和基本环境信息（工作目录等），不继承主对话的完整 system prompt。

工具权限控制

工具权限通过 tools 和 disallowedTools 两个字段控制：

内置的工具包括：

  • 只读：Read、Glob、Grep

九、从 Subagent 到 Agent Teams：进化之路

Subagent 系统虽然强大，但有一个本质限制：Subagent 之间不能直接通信。它们只能向主 Agent 汇报结果，无法互相讨论和协作。

这就引出了 Claude Code 的下一个进化形态——Agent Teams（实验性功能）。

Subagent vs Agent Teams 对比

维度 Subagent Agent Teams
上下文 独立窗口，结果回传 独立窗口，完全独立
通信 只能向主 Agent 汇报 队友之间直接消息
协调 主 Agent 管理所有工作 共享任务列表，自主协调
适用 聚焦任务，只需要结果 需要讨论和协作的复杂工作
成本 较低 较高（每个队友是独立实例）

十、总结与最佳实践

快速上手清单

  1. ✅ 创建 .claude/agents/ 目录
  2. ✅ 从六大经典模板中挑选需要的角色
  3. ✅ 根据项目特点定制 description
  4. ✅ 合理分配模型（Haiku/Sonnet/Opus）
  5. ✅ 严格控制工具权限（最小权限原则）
  6. ✅ 将 Agent 配置提交到 Git，团队共享

设计原则

  • description 要精确：Claude 靠它决定何时委派，模糊的描述 = 混乱的委派
  • 工具权限最小化：审查类 Agent 不要给写入权限
  • 善用 Haiku：不需要深度推理的任务，用 Haiku 省钱又快
  • 一个 Agent 一个职责：不要做”万能 Agent”，那还不如直接用主模型

展望

从单一的 AI 助手，到可定义人格的 Subagent，再到多 Agent 团队协作——Claude Code 正在走向AI 工程团队的愿景。

当你的项目有了架构师 Agent 把关设计、审查员 Agent 守护质量、安全 Agent 扫描漏洞、DevOps Agent 管理部署……你拥有的不再是一个工具，而是一个AI 驱动的工程团队。

而这，才刚刚开始。
