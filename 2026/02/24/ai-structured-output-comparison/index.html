别再无脑用JSON了：AI结构化输出的七种武器

几乎所有和大模型打交道的开发者都会面对同一个问题：怎么让LLM输出结构化数据？

十有八九，答案是JSON。毕竟JSON是Web时代的通用语言，所有主流LLM API都原生支持，生态工具链成熟到几乎不用动脑子。

但如果我告诉你，**强制LLM输出JSON会让它的推理能力下降10-15%**，你还会这么笃定吗？

如果我再告诉你，有一种2025年新出的格式能省40%的token成本，而且LLM理解得更准确呢？

这篇文章，我们来一次彻底的清算——把AI结构化输出的七种主流格式拉出来，用真实数据做一场擂台赛。

一、JSON的隐藏代价：你以为的最优解，可能在拖后腿

JSON统治结构化数据交换已经二十年了。在LLM领域，它更是默认选择——OpenAI的Structured Outputs、Anthropic的tool_use、Gemini的responseSchema，全部围绕JSON构建。

但学术界的研究揭示了一个令人不安的事实。

推理性能下降：被格式绑架的大脑

2024年，Tam等人发表了一篇重要论文 **”Let Me Speak Freely?”**，系统测试了强制格式输出对LLM推理能力的影响。结论触目惊心：

当LLM被强制要求以JSON格式输出时，其推理准确率平均下降10-15%。

这不难理解。想象你正在解一道复杂的数学题，但有人要求你每写一步都必须填进一个Excel表格里——你的注意力会被格式消耗，真正用于思考的”算力”就少了。

LLM也一样。当模型需要同时”思考答案”和”维护JSON语法正确性”（匹配花括号、转义引号、保持逗号不多不少），它的推理资源就被分散了。

Token成本：看不见的账单

JSON的另一个问题是冗余。看这个简单的例子：

每条记录都要重复"id"、"name"、"role"这些key，加上花括号、引号、逗号——对机器来说是必要的语法糖，对token账单来说是真金白银的浪费。

100K条记录的JSON大约消耗4.5M tokens。同样的数据用TOON格式？2.7M tokens。省了40%。

按GPT-4o的价格算，这差距在大规模应用中可以是每月数千美元的区别。

二、七种武器：逐一过招

1. JSON — 万金油，但不是银弹

优点：

  • 生态最成熟，所有编程语言都有原生解析器
  • 所有主流LLM API原生支持
  • JSON Schema提供严格的结构验证
  • OpenAI的Structured Outputs号称100%合规率

缺点：

  • 语法冗余（引号、花括号、逗号），token消耗高
  • 强制输出降低推理性能（10-15%）
  • 深层嵌套时可读性急剧下降

最佳场景： API交互、跨语言数据交换、需要严格Schema验证的生产环境。

一句话评价： 不是最好的，但最不容易出错的。

2. YAML — LLM的心头好

优点：

  • 可读性最好，人类和LLM都爱
  • token比JSON少**25-30%**（没有引号和花括号）
  • 在ImprovingAgents（2025）的1000题嵌套数据测试中，GPT-5 Nano用YAML的准确率达到**62.1%**，远超JSON的50.3%

缺点：

  • 缩进敏感——LLM生成时经常搞错缩进，一个空格之差就是语法错误
  • 没有原生Schema约束机制
  • 安全隐患：Python的yaml.load()可以执行任意代码（必须用yaml.safe_load()）

研究数据： GPT-5 Nano用YAML比XML准确率高17.7个百分点，是所有格式中表现最好的。

最佳场景： 配置文件、人机混合编辑场景、作为LLM的输入格式（而非输出）。

一句话评价： 读起来最舒服，但让LLM写YAML就像让小学生写诗——意思对了，格式经常歪。

3. TOON — 专为LLM打造的新物种

2025年底横空出世的Token-Oriented Object Notation，可能是近年来最有意思的格式创新。

核心思路极其简洁：Schema声明一次，数据用CSV风格排列。

对比JSON版本：

TOON的设计哲学是：key只写一次（在Schema声明里），数据行只有值——彻底消灭重复。

优点：

  • token比JSON少**40%**（100K记录：4.5M → 2.7M tokens）
  • 显式数组长度声明[N]帮助LLM自我校验（”我应该输出2行”）
  • 聚合任务准确率：TOON 54.4% vs JSON 48.8%
  • 检索任务：TOON 73.9% vs JSON 69.7%（209题测试）

缺点：

  • 只适合均匀的表格数据，深度嵌套时效率下降
  • 2025年底才出现，生态极不成熟
  • 几乎没有现成的解析库

最佳场景： 大批量表格数据的输入、需要极致压缩token成本的场景。

一句话评价： 思路惊艳，但现在用它就像2009年用Node.js——前途光明，生态荒芜。

4. XML — 老将不死

优点：

  • 自描述性强（开闭标签一目了然）
  • DTD/XSD验证体系成熟了二十年
  • Claude偏爱XML格式——Anthropic官方文档推荐用XML标签组织prompt

缺点：

  • token消耗最高——比JSON多14%，比TOON多114%
  • LLM理解准确率最低：在Gemini测试中只有33.8%
  • 开闭标签的冗余在LLM场景下是双重浪费

最佳场景： 文档标记（HTML本质上就是XML变体）、需要混合内容（文本+结构）的场景、遗留系统集成。

一句话评价： 在LLM时代，XML像一位德高望重但略显笨重的老将军——值得尊重，但别派他去跑百米。

5. Markdown — 轻装上阵

优点：

  • token比JSON少**34-38%**，比YAML还少约10%，是所有格式中最省的
  • LLM天然擅长生成Markdown（训练数据中大量存在）
  • 可读性极佳

缺点：

  • 结构表达力弱，没有标准Schema
  • 嵌套深度有限（靠标题层级#，最多6层）
  • 解析不确定性高——同样的内容可以有多种Markdown写法

最佳场景： LLM输入中的层级数据展示、非严格结构化的报告生成。

一句话评价： 最省钱，但也最”松散”——像是用便签纸写合同。

6. TOML — 配置界的瑞士军刀

优点：

  • 语义明确，比YAML更安全（没有代码执行风险）
  • Rust和Python生态的宠儿（Cargo.toml、pyproject.toml）
  • 类型系统比YAML更严格

缺点：

  • 不擅长深度嵌套（超过3层就开始难看）
  • LLM训练数据中占比少，模型”不太熟”
  • 没有任何LLM API原生支持

最佳场景： 应用配置文件，但不适合作为LLM交互格式。

一句话评价： 优秀的配置格式，但在LLM的世界里是个陌生人。

7. CSV/TSV + JSON Lines — 朴素的力量

CSV/TSV优点： 表格数据最省token，几乎零语法开销。
CSV/TSV缺点： 只能表示扁平结构，嵌套数据无能为力。

JSON Lines优点： 每行独立解析，天然适合流式处理和日志。
JSON Lines缺点： 本质还是JSON，单行的token开销没有减少。

最佳场景： 大批量表格数据输入、流式日志处理。

三、实测数据：LLM到底更懂谁？

说了这么多，让数据说话。以下是来自多项研究的实测对比：

ImprovingAgents (2025) — 1000题嵌套数据测试

格式 GPT-5 Nano准确率 Gemini准确率
YAML 62.1% 48.2%
JSON 50.3% 55.7%
XML 44.4% 33.8%

关键发现： 不同模型有不同偏好。GPT系列在YAML上表现最好，Gemini反而更喜欢JSON。这不是格式本身的问题，而是训练数据分布的影响——模型在哪种格式上见过更多高质量数据，就对哪种更擅长。

TOON Benchmarks — 209个检索问题

格式 检索准确率 聚合准确率
TOON 73.9% 54.4%
JSON 69.7% 48.8%

TOON在两项任务上都胜出，尤其在聚合任务（求和、计数、平均值）上优势明显——显式的Schema声明帮助LLM更好地理解数据结构。

“Let Me Speak Freely?” — 格式约束的代价

这项研究的核心发现是：格式越严格，推理越受损。

  • 自由文本输出：基线准确率
  • JSON强制输出：准确率下降10-15%
  • 两阶段法（先自由推理再格式化）：准确率恢复甚至提升

这引出了一个重要的最佳实践——两阶段法。

四、各大模型厂商的结构化输出方案

OpenAI — 最完善的方案

OpenAI提供了目前最成熟的结构化输出支持：

  • response_format：指定json_object或json_schema
  • Structured Outputs：基于JSON Schema的约束解码，gpt-4o评测达到100%合规率
  • 实现原理：在token生成时，通过CFG（上下文无关文法）约束采样空间，确保输出一定是合法JSON

Anthropic — Claude的XML情结

Anthropic走了一条不同的路：

  • 通过tool_use（工具调用）实现结构化输出
  • Claude在内部对XML格式有偏好——Anthropic官方推荐用XML标签组织复杂prompt
  • 没有像OpenAI那样的原生JSON Schema约束

这意味着如果你用Claude，最好的策略是：输入用XML组织，输出走tool_use拿JSON。

Google Gemini — responseSchema

  • 通过responseSchema参数指定输出结构
  • Gemini对JSON的理解准确率在多项测试中表现稳定
  • 支持枚举类型和嵌套对象

Mistral — json_schema

  • 提供json_schema参数
  • 功能上类似OpenAI的方案，但生态工具链还在完善中

本地模型 — Grammar-based Constrained Decoding

llama.cpp和vLLM等本地推理框架提供了最灵活的方案：

  • JSON Schema：和云端API类似
  • 正则表达式：用正则定义输出格式
  • CFG（上下文无关文法）：最强大，可以定义任意结构化格式

这意味着本地模型理论上可以支持任何格式的约束输出——包括YAML、TOML、甚至TOON。

五、最佳实践：”两阶段法”和场景选择指南

两阶段法：让LLM先想清楚再排版

腾讯云的一篇技术文章提出了**”NL-to-Format”两阶段法**，效果惊人：

第一阶段： 让LLM用自由自然语言回答问题，不施加任何格式约束。
第二阶段： 把第一阶段的自由文本作为输入，让LLM（或同一个模型的第二次调用）将其转换为目标格式。

结果： 正确率提升**44.8%**。

这背后的逻辑很简单：人类也是先想清楚再写报告，而不是一边想一边填表格。给LLM同样的”思考自由”，它就能表现得更好。

代价是两次API调用，但如果准确率提升44.8%，这个代价在大多数场景下都值得。

场景选择指南

输入数据（喂给LLM的）：

  • 表格数据 → TOON（省40% token）
  • 嵌套配置 → YAML（可读性最好）
  • 通用场景 → JSON（最安全）
  • 文档混合 → XML（Claude场景）或Markdown

输出数据（让LLM生成的）：

  • 需要严格保证 → JSON + 约束解码（100%合规）
  • 需要最佳推理 → 两阶段法（先自由输出再格式化）
  • 流式处理 → JSON Lines

Token成本速算

假设你有10万条用户记录需要喂给LLM：

格式 预估Token数 相对成本
CSV ~1.8M 40%
Markdown ~2.3M 51%
TOON ~2.7M 60%
YAML ~3.3M 73%
JSON ~4.5M 100%
XML ~5.1M 114%

选对格式，token账单可以直接砍半。

六、决策树：你该用哪个？

做选择其实不难，问自己三个问题：

问题1：你是在给LLM喂数据，还是让LLM吐数据？

  • 喂数据（输入）→ 往下看
  • 吐数据（输出）→ JSON + 约束解码，除非你有特殊需求

问题2（输入场景）：你的数据是什么形状？

  • 扁平表格 → TOON 或 CSV（最省token）
  • 嵌套结构 → YAML（最好读）或 JSON（最安全）
  • 文档+结构混合 → XML 或 Markdown

问题3：你的准确率要求有多高？

  • 必须100%正确 → JSON + OpenAI Structured Outputs
  • 允许一定容错 → 两阶段法（先自由推理再格式化）
  • 成本优先 → TOON/Markdown输入 + 最小化输出

总结

JSON依然是AI结构化输出的”安全选择”——生态成熟、厂商支持全面、约束解码可以保证100%合规。但它不是唯一的选择，更不是所有场景的最优解。

三个核心takeaway：

  1. 输入用轻量格式：TOON省40%成本，YAML省25-30%，别把钱浪费在重复的key上
  2. 输出用JSON+约束解码：这是目前唯一能保证格式100%正确的方案
  3. 复杂推理用两阶段法：先让LLM自由思考，再格式化输出，准确率提升44.8%

格式之争的本质不是”谁最好”，而是**”谁最适合你的场景”**。希望这篇七种武器的对比，能帮你在下次设计LLM pipeline时，做出更聪明的选择。
