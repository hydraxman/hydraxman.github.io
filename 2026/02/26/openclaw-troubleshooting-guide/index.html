前言

OpenClaw 最近彻底火了（36氪头条都在写”谷歌连夜封禁 OpenClaw”），越来越多人开始上手部署自己的 AI 助手。但安装和运行过程中，总会遇到各种各样的问题。

我自己从 2026 年初开始使用 OpenClaw，踩过无数坑，也在社区里看到大量重复的求助帖。总结下来，问题基本就四大类：

  1. 🌐 网络问题（占 60%+）— 国内用户的头号大敌
  2. 🤖 AI 模型请求问题（占 20%）— API Key、模型兼容性、token 限制
  3. 💻 系统兼容性问题（占 10%）— Node.js 版本、系统差异、依赖缺失
  4. 🔄 运行稳定性问题（占 10%）— Gateway 崩溃、内存泄漏、长时间运行不稳定

这篇文章把每一类的常见症状、原因和解决方案都列出来，建议收藏，遇到问题直接 Ctrl+F 搜报错信息。

问题 8：API 速率限制（429 Too Many Requests）

症状：

解决方案：

  • 检查 API 账户的用量和限额
  • 升级 API 计划获得更高额度
  • 减少并发请求：避免同时运行多个高频任务
  • 给搜索类操作加间隔（sleep 2s），特别是 Brave Search 免费版（1 req/s 限制）

问题 9：Token 超限 / 上下文太长

症状：

解决方案：

  • 清理对话历史：长时间对话会积累大量 token
  • 拆分任务到子 Agent（sessions_spawn），每个任务独立上下文
  • 避免在消息中发送 base64 图片数据（这个坑很大！会直接卡死）
  • 减少加载的 Memory/Skill 文件体积

第四类：运行稳定性问题

这是很多人忽略但其实非常重要的一类。OpenClaw Gateway 是一个长时间运行的 Node.js 进程，内存泄漏、WebSocket 断连、消息渠道掉线都是常见问题。

问题 15：Gateway 运行一段时间后卡死 / 不响应

症状：Bot 突然不回消息了，但 openclaw status 显示 Gateway 还在运行。

原因：Node.js 长时间运行可能出现内存泄漏，或 WebSocket 连接静默断开。

解决方案：配置定时重启（强烈推荐！）

这是我实践后总结的最佳策略——每天定时重启 Gateway 5 次：

为什么要定时重启？

  • 预防内存泄漏积累
  • 自动修复静默断连的 WebSocket
  • 刷新消息渠道连接（飞书、Telegram 等）
  • 重启成本极低（1-3 秒），但能大幅提升稳定性

问题 16：配置远程消息重启支持

当你不在电脑旁时，如果 Gateway 出问题，可以通过消息渠道远程重启：

开启后，你可以通过 Telegram/飞书等渠道给 AI 发消息：

“重启 gateway”

AI 助手就会执行 openclaw gateway restart，非常方便。

⚠️ 注意：commands.restart 默认是 false，必须手动开启。这是一个很多人不知道的配置项。

问题 17：Brave Search API 限流导致 Cron 任务失败

症状：定时任务（如每日新闻摘要）连续失败，日志显示 429 错误。

原因：Brave Search 免费版限制 1 请求/秒，连续搜索很容易触发限流。

解决方案：

  • 搜索请求之间加 sleep 2s 间隔
  • 减少单次任务的搜索次数
  • 失败时降级继续（跳过失败的搜索，用已有结果完成任务）
  • 考虑升级为付费版或使用 Tavily 等替代搜索 API

问题 18：飞书重复消息

症状：AI 对同一条消息回复了两次。

原因：飞书消息队列偶尔会重发之前的消息（queue replay）。

解决方案：目前需要手动忽略。OpenClaw 新版本已在改进消息去重逻辑。

综合稳定性方案：我的配置清单

经过几个月的实战，我总结出一套让 OpenClaw 7×24 小时稳定运行的组合拳：

1. 定时重启 Gateway

2. 开启远程重启

3. 使用 LaunchAgent 自动启动（macOS）

确保开机/重启后 Gateway 自动运行。openclaw gateway start 会自动配置 LaunchAgent。

4. 网络保障

  • 代理保持常开，设为全局模式
  • DNS 使用可靠的公共 DNS（223.5.5.5）
  • 如果可能，使用 GitHub Copilot 作为模型提供商（国内延迟最低）

5. 监控检查

6. 及时升级
