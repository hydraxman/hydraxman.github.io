当 AI Agent 真开始维护代码，为什么成绩会从 80％ 掉到 38％？EvoClaw 论文深读

最近看到一篇很对味的论文：EvoClaw: Evaluating AI Agents on Continuous Software Evolution。

它问的不是“模型会不会写代码”，也不是“能不能修一个 issue”，而是一个更接近真实世界的问题：

如果让 AI Agent 像真人工程师一样，连续几周、几个月地在同一个代码库里迭代，它到底行不行？

论文给出的答案很直接：现在还远远不行。

在传统的“单题作答”式评测里，顶级模型/Agent 的分数可以轻松上 80%；但一旦进入连续演化场景，最好成绩也只有 38.03%。这不是简单的“题更难了”，而是现有 Agent 在长期维护、回归控制、技术债管理、上下游依赖协调这些真正的软件工程问题上，暴露出了明显短板。

这篇论文很值得认真读，因为它不只是又造了一个 benchmark，而是在试图把 Agent 评测从“做一道题”推进到“维护一个系统”。如果你在关注 OpenClaw、Claude Code、Codex CLI、OpenHands 这类长期运行 Agent，这篇论文几乎是必读。

一句话总结这篇论文

EvoClaw 的核心贡献，是把 AI Agent 的代码能力评测，从“孤立任务求解”升级成“连续软件演化能力评测”。

作者先做了一个自动化流水线 DeepCommit，从开源仓库真实的 commit 历史中重建出可以执行、可以验证的“里程碑 DAG（Milestone DAG）”；再基于这些里程碑，构造出一个要求 Agent 在同一代码库里持续开发、持续背负历史包袱的 benchmark —— EvoClaw。

结果非常扎眼：

  • 独立任务评测里，模型经常能上 80%+
  • 连续演化评测里，最强配置也只有 38.03%
  • 最好的 Resolve Rate（完整解决率） 也远低于大家对“能干活”的直觉
  • Agent 会写新功能，但越来越不会“守住系统”

换句话说，大家以为 Agent 的问题是“不会写”，其实更大的问题是“不会长期维护”。

这篇论文到底在解决什么问题？

今天大部分 AI 编程 benchmark，评估的都是这种模式：

  1. 给你一个任务
  2. 你改代码
  3. 跑测试
  4. 环境重置
  5. 下一题重新开始

这种评测当然有价值，但它天然绕开了真实开发里最麻烦的部分：

  • 你今天留下的烂接口，会不会坑死下周的需求？
  • 你为了赶进度做的 workaround，会不会在第三个迭代里引发回归？
  • 你加的新功能，是否破坏了原有行为？
  • 你有没有能力在一个不断变脏、不断变复杂的系统里持续推进？

现实世界的软件开发，本质上是一个有历史负担的连续过程。你写的不是“一次性答案”，而是在一个活着的系统上不断叠加决策。

EvoClaw 就是在补这一块。

论文里有个判断我很认同：真正困难的不是局部实现，而是跨任务的系统完整性维护。

DeepCommit：先把“真实软件演化轨迹”重建出来

EvoClaw 之所以有意思，不只是因为它提出了一个新任务，而是因为它的数据构造方式比很多 benchmark 更接近真实工程。

作者先做了一个叫 DeepCommit 的 agentic pipeline，用来从噪声很大的 commit 历史中，重建出“可验证的软件演化路径”。

整个流程大致分三层。

第一层：从 commit 历史里抽结构化信息

论文里提到，他们会先对仓库历史做静态分析，提取包括：

  • source filtering
  • commit extraction
  • PR / Issue 信息
  • release 信息
  • commit DAG
  • code metrics
  • symbol changes

这一步的目的，是把原始、杂乱、粒度很细的 commit 历史，先整理成后续 Agent 能操作的结构化素材。

第二层：让 LLM Agent 重建里程碑 DAG

这是 DeepCommit 最核心的一步。

论文不是直接拿单个 commit 当任务，而是把一串语义上相关的提交，聚成一个更合理的单元：milestone。

作者把 milestone 定义为：围绕一个相对完整开发目标的、语义一致的提交集合。

这很关键。因为：

  • 单个 commit 太碎，噪声太大
  • 整个 release 又太大，任务范围失真
  • milestone 刚好落在一个比较合理的工程粒度上

DeepCommit 里，Agent 会经历四个阶段去构建这个 Milestone DAG：

  • seed discovery
  • milestone consolidation
  • dependency inference
  • milestone decomposition

最终得到的不是一条简单时间线，而是带依赖关系的里程碑图。这比“按时间顺序一个个做”更真实，因为现实开发里很多任务本来就存在前置依赖。

第三层：把运行环境也补齐，让 benchmark 真能跑

很多论文 benchmark 的问题，不在任务定义，而在环境不靠谱。仓库历史一旦回到老版本，经常会遇到：

  • 依赖装不起来
  • 测试收集失败
  • flaky tests 一堆
  • patch 应用顺序不对直接炸

DeepCommit 在这块做得比较实。

论文里写到，Phase 3 会做：

  • testbed construction
  • test collection
  • DAG refinement
  • fallback patches repair
  • flaky test filtering

然后再做质量保证，检查：

  • 里程碑图结构是否有效
  • 环境是否可编译、可执行
  • 测试是否稳定

论文给出的结果是：

  • **平均 test collection success rate 超过 85%**（摘要）
  • 在正文里，test collection rate 达到 87.1%
  • Pass-to-Fail rate 被压到 ≤ 0.026%

这说明作者不是只在“任务设计”上讲故事，而是真的把可执行性做出来了。

EvoClaw benchmark 长什么样？

基于 DeepCommit，作者最终构造了 EvoClaw：

  • 98 个 human-verified milestones
  • 来自 7 条 evolution itineraries
  • 每条 itinerary 对应一个高影响力开源仓库的一个 release range
  • 覆盖 5 种编程语言

这里最重要的不是样本量本身，而是任务组织方式。

传统 benchmark：独立任务模式

在独立模式下，每做完一题，环境就重置。你上一题留下的烂尾设计、脏接口、潜在 bug，下一题都不用负责。

EvoClaw：连续任务模式

EvoClaw 则相反。

它通过一个 dependency-driven scheduler，按照里程碑 DAG 的依赖逐步解锁任务。Agent 始终在一个持续存在的代码库状态上开发：

  • 上一题改的代码会保留下来
  • 上一题埋的坑也会保留下来
  • 下游任务必须建立在上游任务产出的真实状态上

这就是它最像真实软件工程的地方。

论文用了一个很好的描述：这是 develop-in-place, evaluate-in-isolation。

也就是：

  • 开发是在持续演化的真实代码状态里进行
  • 评测是在隔离快照里完成

这样既保留了连续开发的历史约束，又尽量保证评测严格性。

他们怎么评？不是只看“做没做出来”

EvoClaw 不是只看 Agent 有没有把新功能写出来，而是同时看两件事：

1. Recall：新功能实现得够不够完整

这衡量的是“往前推进”的能力。

2. Precision：你有没有把已有系统搞坏

这衡量的是“守住系统”的能力。

作者据此定义了一个综合 Score，同时报告更严格的 Resolve Rate：

  • Score：兼顾新增功能完成度与系统回归控制
  • Resolve Rate：这个 milestone 是否被完整解决

这个设计我觉得很对。

因为真实工程里，只会加功能但总把旧功能搞坏，不算真的完成任务。

最核心的实验结果：连续开发一来，分数直接塌了

论文测试了 4 类 agent framework、多个前沿模型配置：

  1. Claude Code

    • Claude Opus 4.5
    • Claude Sonnet 4.5
    • Claude Opus 4.6
    • Claude Sonnet 4.6

  2. Codex CLI

    • GPT 5.2
    • GPT 5.2-Codex
    • GPT 5.3-Codex

  3. Gemini CLI

    • Gemini 3 Pro
    • Gemini 3.1 Pro
    • Gemini 3 Flash

  4. OpenHands

    • Claude Opus 4.6
    • GPT 5.3-Codex
    • Gemini 3 Flash
    • Kimi K2.5
    • MiniMax M2.5

连续评测下的最好成绩是多少？

论文 Table 2 的最好分数是：

  • OpenHands + Claude Opus 4.6：38.03%
  • Claude Code + Claude Opus 4.6：36.29%
  • Claude Code + Claude Sonnet 4.6：29.58%
  • Codex CLI + GPT 5.3-Codex：28.88%
  • Gemini CLI + Gemini 3 Pro：24.25%

这已经很说明问题了：

今天最强的 Agent 组合，在连续软件演化场景里，最好也就 38 分。

更刺眼的是：独立任务明明很好，连续任务却崩了

论文反复强调一个核心发现：

在独立任务模式下，模型普遍能拿到 80% 以上；但进入连续演化环境后，最好成绩只剩 38.03%。

作者举了一个很有代表性的例子：

  • 在 scikit-learn 上，Claude Sonnet 4.6 的独立评测分数能到 93.2%
  • 但放进连续演化设置后，只剩 21.1%

这个对比很残酷，但也非常真实。

它说明：

当前 Agent 并不是不会做单点任务，而是不会在带历史负担的系统里持续做对。

比低分更关键的发现：Recall 还能涨，Precision 很快见顶

这是我觉得整篇论文最重要的洞察。

论文第 5.4 节发现，在连续软件演化中，Agent 表现出一种很明显的不对称：

  • Recall 基本还能接近线性增长
  • Precision 会很快饱和

翻译成人话就是：

  • Agent 还是能不断往系统里“加东西”
  • 但它越来越守不住已有系统的正确性

所以问题不是“它不会写功能”，而是“它不会维护系统”。

这跟很多一线使用体验其实高度一致：

  • 第一轮改动看起来很顺
  • 第二轮开始出现边角回归
  • 第三轮需求一来，旧逻辑开始互相污染
  • 再往后，Agent 越修越乱，最后只能人工接管

论文用饱和曲线去拟合 cumulative score trajectory，结论也很明确：

  • 独立评测下，能力增长更接近线性
  • 连续评测下，分数会出现明显 ceiling
  • 随着演化推进，这个 gap 会持续拉大

这非常像现实里的“系统熵增”。

真正拖垮 Agent 的，不是单题难度，而是错误传播

论文第 5.5 节用了一个很形象的说法：snowball effect。

意思是，一旦前面某个 milestone 留下回归或设计问题，这些问题不会停留在局部，而是会沿依赖链一路向下污染后续任务。

最后就会出现这种情况：

  • 后面的 Agent 明明还能写新代码
  • 但它的修改必须建立在一个已经被污染的系统状态上
  • 于是每往前一步，排查和修补历史问题的成本都在上升
  • 最终整体开发被拖停

这也是为什么论文里会看到：

  • Score 和 Resolve Rate 之间有很大落差
  • 很多 milestone 能做到“部分进展”
  • 但很少能“完整解决”
  • 被完整解决的往往还是早期、依赖少的 milestone

这其实非常像真实项目：越到后期，最难的不是开发新功能，而是和历史兼容。

行为层面的分析也很有意思：好 Agent 会主动探索和验证

论文第 5.6 节没有停留在分数层，而是进一步分析行为模式。

结论也很接地气：

成功概率更高的行为

  • 主动探索代码库
  • 在动手前先理解上下文
  • 经常做验证和测试确认
  • 保持对已有结构的尊重

更容易失败的行为

  • 盲目 trial-and-error
  • 看到报错就局部打补丁
  • 缺乏系统性 verification
  • 不理解架构就直接改 API 或职责边界

论文还提到一些典型代码异味，例如：

  • responsibility boundary misplacement
  • shotgun fixes
  • API signature degradation

说白了就是：

Agent 在局部修复时，经常会做出“短期看似有效、长期一定埋雷”的工程决策。

这不是 token 不够的问题，而是“软件演化意识”还不够强。

这篇论文对 OpenClaw / 长周期 Agent 有什么启发？

我觉得这篇论文的价值，不只是 benchmark 本身，而是它把一个经常被忽略的问题彻底摊开了：

如果 Agent 要在真实世界长期工作，它必须把“软件维护能力”当成核心能力，而不是附属能力。

论文在引言里甚至直接点名了 OpenClaw 这一类长运行 Agent 场景：

  • Agent 会持续接收用户新需求
  • 会不断扩写自己的工具接口
  • 会在动态环境里调整已有能力
  • 它写下来的代码，不是一次性产物，而是未来自己的工作底座

这就意味着，未来评估 Agent，不能只看：

  • 单题 pass rate
  • issue resolution rate
  • 一次性 repo generation 成功率

还得看：

  • 它能不能在多轮迭代后保持系统健康
  • 它会不会制造难以发现的技术债
  • 它是否具备 regression-aware 的开发习惯
  • 它是否知道什么时候该重构、什么时候该止损

从这个角度看，EvoClaw 很像是在提醒整个行业：

Agent 的下一个瓶颈，不是写出第一版代码，而是维护第十版代码。

我对这篇论文的几个判断

1. 这是一个方向非常对的 benchmark

过去很多 benchmark 本质上都在考“静态解题能力”。

EvoClaw 往前走了一步，开始考：

  • 历史依赖
  • 累积误差
  • 回归控制
  • 连续演化

这几个维度比“会不会写一段功能代码”更接近软件工程真实难点。

2. 它揭示了 Agent 现阶段最脆弱的地方

论文最有价值的，不是证明某个模型第一，而是揭示出一个普遍现象：

现有前沿 Agent 普遍缺乏长期维护能力。

谁第一其实不关键，关键是第一名也只有 38.03%。

这意味着整个赛道都还没真正解决“持续开发”的问题。

3. 未来优化重点应该从“更会写”转向“更会守”

如果论文判断成立，那么下一阶段真正有效的优化方向，可能不只是更强模型本身，而是：

  • 更强的状态管理
  • 更强的架构记忆
  • 更严格的回归防线
  • 更主动的测试策略
  • 更好的代码库探索机制
  • 更成熟的 stop / reflect / verify 流程

也就是说，Agent 的工程 scaffold 设计，未来可能和 base model 本身一样重要。

4. 这个 benchmark 也有现实限制

当然，EvoClaw 也不是没有边界。

比如我觉得它至少还有几个限制：

  • 数据规模目前不算大，核心 benchmark 是 98 个 milestones / 7 条 itinerary
  • 评测成本高，论文提到一次完整评测对前沿模型大约要 500 美元
  • 它重点评的是“连续实现与维护”，并不覆盖更高层的产品规划与需求协商
  • benchmark 仍然来自历史开源演化轨迹，和真实线上系统的混沌环境相比，复杂度依旧是收敛过的

但这些不构成否定，反而说明这个方向值得继续做大。

最后的结论

如果只看传统 benchmark，你很容易产生一种错觉：

AI Agent 已经越来越像“能独立写代码的软件工程师”了。

但 EvoClaw 告诉我们，真相没那么乐观。

Agent 在“单题完成”这件事上确实进步很快；可一旦进入真实的软件演化场景，它马上会碰到三个硬问题：

  • 历史决策会反噬未来任务
  • 回归控制远比生成功能更难
  • 错误会沿依赖链不断累积并放大

所以，这篇论文真正戳中的不是某个模型的短板，而是整个 Agent 评测范式的盲区。

未来真正能落地的 Agent，不只是会写代码的 Agent，而是能长期维护自己代码资产的 Agent。

从这个意义上说，EvoClaw 不只是一个 benchmark，它更像是一张“行业体检报告”。

而这份体检报告的结论，目前并不轻松。

  9. 行为层面的分析也很有意思：好 Agent 会主动探索和验证
    1. 成功概率更高的行为
    2. 更容易失败的行为
  10. 这篇论文对 OpenClaw / 长周期 Agent 有什么启发？
  11. 我对这篇论文的几个判断
    1. 1. 这是一个方向非常对的 benchmark
    2. 2. 它揭示了 Agent 现阶段最脆弱的地方
    3. 3. 未来优化重点应该从“更会写”转向“更会守”
    4. 4. 这个 benchmark 也有现实限制
