Agent不等于模型：LangChain万字长文拆解「套壳工程」的本质

Agent = Model + Harness。 如果你不是模型本身，那你就是 Harness 的一部分。

这句话出自 LangChain 最新发布的博客 The Anatomy of an Agent Harness。短短一句话，把整个 AI Agent 行业最核心的分工讲透了。

什么是 Harness？直译是「套具」——马匹身上的挽具、降落伞的吊带。在 AI Agent 语境下，Harness 是围绕模型搭建的一切工程系统：系统提示词、工具集、沙箱环境、编排逻辑、记忆管理、上下文压缩……模型提供智能，Harness 让智能变得有用。

这篇文章不是简单翻译。我会结合 Claude Code 刚泄露的源码（1884个TypeScript文件），用真实代码验证 LangChain 提出的每一个 Harness 组件，看看理论和实践之间到底有多大差距。

一、为什么需要 Harness？

先回到最基本的问题：大语言模型能做什么？

答案很简单——接收文本/图片/音视频，输出文本。就这样。

它不能：

  • 跨对话保持记忆
  • 执行代码
  • 访问实时信息
  • 自己搭环境装依赖

所有这些能力，都需要 Harness 来提供。

举个最简单的例子：我们日常用的「聊天」功能，本质上就是一个 while 循环——Harness 追踪历史消息，拼接新输入，喂给模型。没有这个循环，模型每次都是失忆的。

LangChain 的核心观点是：我们应该从「期望 Agent 做什么」反推「Harness 需要什么功能」。

这个思维方式很重要。不是先堆功能再看能干什么，而是先定义行为目标，再设计工程系统。

二、七大 Harness 组件

LangChain 将 Harness 分解为七大核心组件。下面逐一拆解，并用 Claude Code 源码做实证。

1. 文件系统：最基础的原语

期望行为： Agent 需要持久存储，能操作真实数据，跨会话保持工作状态。

Harness 方案： 提供文件系统抽象和操作工具。

LangChain 把文件系统称为「最基础的 Harness 原语」，因为它解锁了几乎所有其他能力：

  • 工作空间：Agent 可以读数据、代码、文档
  • 增量输出：不用把所有东西塞进上下文，可以写文件慢慢来
  • 协作表面：多个 Agent 和人类通过共享文件协调工作

加上 Git，文件系统就有了版本控制——Agent 可以追踪进度、回滚错误、分支实验。

Claude Code 验证： tools/ 目录有专门的 FileReadTool、FileWriteTool、FileEditTool，而且 Dream System 的记忆整理完全基于文件系统——MEMORY.md 就是 Agent 的长期记忆文件，每次启动自动注入上下文。

2. Bash + 代码执行：万能工具

期望行为： Agent 应该能自主解决问题，而不需要人类预先为每种操作设计工具。

Harness 方案： 提供 Bash 工具，让模型通过写代码自主解决问题。

传统的 Agent 执行是 ReAct 循环——推理、调用工具、观察结果、循环。但 Harness 只能执行它预置的工具。与其为每种操作都写一个工具，不如给 Agent 一个万能工具——Bash。

模型可以即时编写代码来创造自己的工具，而不是被限制在预配置的固定工具集里。

Claude Code 验证： BashTool 是 Claude Code 使用最频繁的工具，支持交互式命令、超时控制、输出截断。更有趣的是，源码里 Bash 被分为不同权限级别——有些命令（如 rm -rf）需要用户确认，而 ls、cat 等只读命令则自动放行。

3. 沙箱与验证：安全执行环境

期望行为： Agent 需要安全的环境来执行代码、观察结果、验证工作。

Harness 方案： 沙箱隔离 + 预装工具链 + 自验证循环。

让 Agent 在本地直接跑代码？太危险了。沙箱提供了隔离的执行环境——可以按需创建、并行扇出、用完销毁。

好的 Harness 还会预装工具链：语言运行时、Git CLI、浏览器、测试框架。这让 Agent 能形成自验证循环——写代码 → 跑测试 → 检查日志 → 修 bug → 再跑测试。

Claude Code 验证： 这正是 Verification Agent 的工作方式。它的 Prompt 里明确写道：

「每个 PASS 必须附带实际执行的命令和输出。没有 Command run 的检查不算 PASS，只算 Skip。」

不是看代码猜结论，而是必须执行才算验证。Claude Code 的 ULTRAPLAN 模式甚至把沙箱搬到了云端——启动远程容器，运行 Opus 级模型思考 30 分钟。

4. 记忆与搜索：持续学习

期望行为： Agent 应该记住它见过的东西，并能访问训练截止后的新信息。

Harness 方案： 文件系统记忆（AGENTS.md）+ Web 搜索 + MCP 工具。

模型的知识止步于训练数据截止日期。Harness 通过两种方式补救：

长期记忆： 通过 AGENTS.md（或 MEMORY.md）这样的标准文件，Agent 在每次启动时自动加载。Agent 在工作中更新这个文件，Harness 在下次启动时注入更新后的版本。这是一种持续学习——跨会话积累知识。

实时知识： Web Search 和 MCP 工具让 Agent 能访问训练截止后的新信息——新版本的库文档、最新的 API 变更、当前的数据。

Claude Code 验证： Dream System 就是记忆 Harness 的终极形态。它不是被动等 Agent 写记忆，而是主动整理——模拟 REM 睡眠，四个阶段（定位 → 采集 → 整合 → 修剪），把零散对话提炼成结构化知识。比 LangChain 描述的 AGENTS.md 方案高级一个量级。

5. 对抗上下文腐烂

期望行为： Agent 的性能不应该随着工作时间增长而下降。

Harness 方案： 上下文压缩 + 工具输出截断 + Skills 渐进式暴露。

上下文腐烂（Context Rot）是一个被严重低估的问题——随着上下文窗口填满，模型的推理能力和任务完成能力会显著下降。上下文是稀缺资源，Harness 必须精心管理。

LangChain 提出三种策略：

策略 方法 解决什么问题
Compaction 智能摘要+卸载旧上下文 上下文快满了
工具输出截断 保留头尾、存文件 大输出占太多上下文
Skills 渐进暴露 只加载元数据，按需展开 工具太多挤占上下文

LangChain 说了一句非常精辟的话：

今天的 Harness 本质上是优质上下文的投递系统。

不是给模型更多信息，而是在正确的时间给正确的信息。

Claude Code 验证： 源码中 systemPromptManager 管理着 110+ 条碎片化 Prompt 片段，根据当前环境（操作系统、工作模式、任务类型）动态拼接——macOS 上拼 macOS 的路径约定，调试模式加调试指令，Git 操作加安全提示。这就是「按需暴露」的极致实践。

6. 长时间自主执行

期望行为： Agent 应该能自主完成复杂任务，跨多个上下文窗口持续工作。

Harness 方案： 文件系统 + Git + Ralph Loop + 规划 + 自验证。

这是所有 Harness 组件开始复合发力的地方。

Ralph Loop 是一个很有意思的 Harness 模式：当模型试图退出时，Harness 通过 Hook 拦截退出信号，在一个干净的上下文窗口里重新注入原始任务目标，强迫 Agent 继续工作。文件系统让这成为可能——每次迭代从头开始，但从上次迭代的文件状态接续。

配合规划（在文件系统里维护计划文件）和自验证（每一步都跑测试），Agent 就能跨越多个上下文窗口持续推进任务。

Claude Code 验证： Coordinator Mode 是 Long Horizon 执行的工业级实现。它不是一个 Agent 硬撑，而是调度器+多 Worker 协作：

  1. 并行派出 Research Workers 调查代码库
  2. Coordinator 综合调研结果，制定方案
  3. 派出 Implementation Workers 改代码
  4. 派出 Verification Workers 验证改动

每个 Worker 有独立上下文，通过 Scratchpad 共享知识。这比 Ralph Loop 的「单 Agent 循环」高明得多。

7. 模型训练与 Harness 的共同进化

LangChain 提出了一个深刻的观察：今天的 Agent 产品（Claude Code、Codex）在后训练时就把 Harness 放在了训练循环里。

模型不是泛泛地学「如何编程」，而是学「如何在特定 Harness 里编程」——怎么用文件系统、怎么执行 Bash、怎么规划、怎么并行分发子 Agent。

这创造了一个飞轮：发现有用的 Harness 原语 → 加入 Harness → 用于训练下一代模型 → 模型在这个 Harness 里更强 → 发现新原语……

但这也带来了一个副作用——过拟合。模型对训练时用的 Harness 非常熟悉，换一种 Harness 反而表现下降。LangChain 举了个例子：Codex-5.3 的文件编辑工具 apply_patch 有特定格式，换成其他 patch 方式性能就掉。理论上一个「真正智能」的模型不应该在乎 patch 格式，但训练的耦合导致了这种脆弱性。

有趣的反例是：Terminal Bench 2.0 排行榜上，同一个 Opus 4.6 模型，在 Claude Code 原装 Harness 里的得分远低于某些第三方 Harness。LangChain 自己的 Agent 就通过只改 Harness 从 Top 30 冲到了 Top 5。

这说明：Harness 工程的优化空间远比我们想象的大。

三、LangChain vs Claude Code：理论与实践对照

维度 LangChain 理论框架 Claude Code 实际实现
文件系统 最基础的原语 ✅ FileRead/Write/Edit + MEMORY.md
Bash 执行 万能工具 ✅ BashTool + 权限分级
沙箱 隔离执行环境 ✅ ULTRAPLAN 远程容器
记忆 AGENTS.md + Web Search ✅✅ Dream System（主动记忆整理）
上下文管理 Compaction + 截断 + Skills ✅✅ 110+ 动态 Prompt 片段
长时间执行 Ralph Loop ✅✅ Coordinator + Multi-Worker
安全 命令白名单 + 网络隔离 ✅✅ 四层权限 + YOLO分类器 + 路径遍历防护
情感连接 （未提及） ✅ Buddy 电子宠物系统
隐身模式 （未提及） ✅ Undercover Mode

Claude Code 在几乎每个维度上都超越了 LangChain 的理论框架。但 LangChain 的贡献在于系统化地定义了 Harness 的概念和分类法——这是整个行业第一次有人把「Agent 套壳」上升为一个正式的工程学科。

四、三个深度洞察

洞察一：Harness 是新的操作系统

如果你仔细看 Harness 的组件列表——文件系统、进程管理（Agent调度）、权限控制、内存管理（上下文）、设备驱动（工具/MCP）——这不就是操作系统的抽象层吗？

模型是 CPU，Harness 是操作系统。

历史总是押韵。1960年代，人们发现裸机太难用，于是发明了操作系统。2020年代，人们发现裸模型太难用，于是发明了 Harness。两者面对的核心问题惊人地相似——如何让底层的计算能力变成用户可用的生产力。

洞察二：Context Rot 是 Agent 时代的内存泄漏

在传统编程中，内存泄漏是程序运行时间越长越慢的根源。在 Agent 时代，上下文腐烂扮演着完全相同的角色——Agent 工作越久，上下文越满，性能越差。

Compaction 就是垃圾回收。Skills 渐进暴露就是动态加载。工具输出截断就是缓冲区管理。

所有传统操作系统解决的问题，Harness 都在重新解决一遍。只不过这次管理的不是比特，而是 Token。

洞察三：Harness 的终局是消失

LangChain 在文末暗示了一个趋势：随着模型越来越强，Harness 中「弥补模型缺陷」的部分会逐渐被模型本身吸收。模型会原生学会规划、自验证、长时间连贯性。

但「围绕模型构建工程系统」的部分永远不会消失——就像你的 CPU 再强，也需要操作系统。文件系统、安全隔离、工具集成这些不是模型的缺陷，是工程架构的必要层。

最终，Harness 工程师会变成 AI 时代的系统架构师。

五、对开发者的启示

如果你在构建 AI Agent，LangChain 这篇文章的方法论值得借鉴：

  1. 从行为反推设计：先定义「我希望 Agent 做什么」，再推导需要什么 Harness 组件
  2. 文件系统优先：文件系统是最被低估的 Harness 原语，几乎所有高级功能都依赖它
  3. 重视上下文管理：Context Rot 是沉默的性能杀手，Compaction 和渐进暴露是必修课
  4. 不要迷信原装 Harness：模型在特定 Harness 上训练不代表那是最优 Harness，Terminal Bench 排行榜已经证明了这一点
  5. 安全不是事后补丁：权限系统、沙箱隔离、路径防护应该从第一天就设计进 Harness
