Claude Code 源码泄露后，我看到了 AI Agent 真正的护城河

源码可以泄露，工程飞轮不会。Claude Code 真正值得研究的，不是某个文件怎么写，而是 Anthropic 如何用 Harness Engineering 把 AI Agent 变成一套可持续交付的软件生产系统。

前言：Claude Code 泄露后，真正的问题才刚开始

前一段时间，Claude Code 的源码因为 npm 包中的 source map 文件泄露到了互联网上。公开报道显示，@anthropic-ai/claude-code 某个版本的 npm 包中包含了 cli.js.map，外界得以重建大量可读的 TypeScript 源码、系统提示词、上下文管理逻辑和 agent orchestration 结构。

很多人的第一反应是：这是不是意味着 Claude Code 的秘密被看光了？

我的判断恰恰相反：泄露出来的只是源码，没有泄露的是 Anthropic 真正的护城河。

对于正在做 Coding Agent、IDE Agent、软件工程自动化的团队来说，源码当然有学习价值。它能告诉我们一个一线 AI 公司如何包装模型、如何组织工具调用、如何管理上下文、如何设计命令行体验。

但这些仍然只是结果，不是动力系统。

真正决定 Claude Code 能否持续进化的，不是某个函数、某段 prompt、某个 orchestration trick，而是背后那套把需求、规划、约束、测试、评估、发布、反馈连接起来的工程飞轮。用今天更前沿的说法，这就是 Harness Engineering。

真正没有泄露出来的是：

  • Anthropic 如何定义需求？
  • 如何组织 agentic engineering？
  • 如何让 Claude Code 以周为单位迭代 feature？
  • 如何设计评估、反馈、测试、发布、回滚这一整套工程飞轮？
  • 如何把模型能力转化成持续的软件产品增长？

换句话说，泄露的是飞机的零件图，不是那套让飞机持续加速、持续迭代、持续升级的 driving force。

这正是 AI 时代软件工程最值得研究的问题：驱动软件向前的到底是什么？

过去几十年，软件工程不断提出各种“Driven Development”：TDD（Test-Driven Development）、BDD（Behavior-Driven Development）、DDD（Domain-Driven Design）、FDD（Feature-Driven Development）、ATDD（Acceptance Test-Driven Development）、MDD（Model-Driven Development）、SDD（Specification-Driven Development）……每一种方法论，本质上都在回答同一个问题：

当复杂性超过个人直觉时，软件开发应该由什么来驱动？

到了 AI Agent 时代，这个问题变得更尖锐。

AI 可以写代码，但它必须被什么东西驱动、约束、校验和牵引。否则，它只是一个高速但无方向的生成器。

从 XDD 到 Harness Engineering：开发的驱动力在迁移

不同的 XDD 方法论，其实是在把软件开发的“方向盘”交给不同的 artifact。

方法论 驱动力 核心 artifact 对 AI Agent 的意义
TDD 测试 单元测试、集成测试 给 Agent 一个可自动验证的目标
BDD 行为 Given-When-Then 场景 把用户行为变成可执行语义
ATDD 验收标准 Acceptance Criteria 把业务成功定义前置
FDD 功能 Feature List / Feature Plan 按可交付功能切片推进
DDD 领域模型 Ubiquitous Language / Domain Model 降低复杂业务中的语义漂移
MDD 模型 UML / DSL / Model 从模型生成或约束实现
SDD 规格 Spec / Plan / Tasks 让规格成为代码生成的源头
PDD / Vibe Coding Prompt 自然语言意图 速度快，但如果缺少约束容易失控

GitHub 在 Spec Kit 文档中对 SDD 的表述很直接：当 specification 和 implementation plan 能够生成代码时，开发中不再是“文档到实现”的松散翻译，而是从 specification 到 implementation 的结构化转换。它强调通过模板、checklist、test-first thinking、constitutional compliance 等方式约束 LLM，使生成代码更可维护、可测试、架构上更稳定。

这与 OpenAI 在 2026 年提出的 Harness Engineering 很接近。OpenAI 在一篇工程博客里总结了一个极端实验：团队在五个月里用 Codex 构建内部 beta 产品，原则是“0 行人工手写代码”。他们提到：

Humans steer. Agents execute.

这句话非常关键。人类不再主要通过手写代码创造系统，而是通过设计环境、脚手架、约束、文档、反馈回路和控制系统来“驾驶”agent。

Anthropic 在自己的 harness design 文章里也表达了类似思想：frontier agentic coding 的性能，不只来自更强的模型，也来自更好的 harness 设计。它们用类似 GAN 的结构，把 generator、evaluator、planner 分开，让生成者负责创造，让评估者负责批判，让规划者负责把模糊需求拆成更完整的产品规格。

这篇文章里有两个特别值得注意的观察。

第一是 Context Anxiety（上下文焦虑）。当长期任务推进到后半程，代码、日志、对话历史不断挤满上下文窗口，Agent 会开始“害怕没地方写了”，于是提前收尾，哪怕任务还没真正完成。Anthropic 的经验是：对某些模型来说，简单压缩上下文并不够，反而需要通过结构化交接文件和 context reset，让新的 Agent 带着清晰任务继续工作。

第二是 Self-Evaluation Bias（自我评估偏差）。让一个 Agent 评价自己刚写出的代码，它往往会过于乐观。尤其是前端设计、产品体验、架构质量这种没有单一单元测试能裁决的问题，自评几乎天然会变成“看起来不错”。所以 Anthropic 的关键做法不是要求同一个 Agent 更自律，而是把生成者和评估者拆开：Generator 负责实现，Evaluator 负责用 Playwright 打开页面、点击交互、截图、验证功能，并从外部给出批判。

这说明：AI 时代的软件工程，正在从“代码驱动”迁移到 harness 驱动。

而 harness 的本质，不只是工具调用封装，而是一组能让 AI Agent 安全、高速、可验证地工作的约束系统：它决定谁规划、谁实现、谁评估，什么时候重置上下文，哪些标准必须显式写进 prompt，哪些结果必须通过外部工具验证。

一个矛盾的现实：Demo 很容易，复杂项目很难

2025 年以来，AI Coding Agent 已经让开发者的生产力飞跃。Cursor、Claude Code、Codex、Copilot……这些工具在写小程序、构建 Demo 时表现惊艳。几十行、几百行、甚至几千行代码的项目，AI Agent 往往可以一气呵成。

原因很简单：当整个项目的代码量没有超过 LLM 的上下文窗口时，AI 对全局了然于胸。

但真实世界的软件开发远非 Demo 那么简单。

当你在一个拥有几十万行代码的成熟产品上工作时，情况完全不同：

  • 上下文爆炸：项目代码量远超 LLM 的上下文窗口
  • 约束叠加：产品已有的架构、设计规范、安全要求、兼容性承诺层层堆叠
  • 增量开发：你不是从零开始，而是在一个精密运转的系统上做“手术”——增加一个 feature、优化一个体验、调整一个内部架构

这就像让一位天才建筑师在一张白纸上画出一栋楼很容易，但让他在一座已经住满人的大厦里加盖一层，难度完全不同。

学术界也在快速研究这个变化。SWE-agent 论文提出 Agent-Computer Interface 的重要性：不是只把 LLM 接到 shell 上，而是要设计适合 agent 操作计算机、观察反馈、修改文件、运行测试的界面。2025 年关于 AI teammates / SE 3.0 的研究，则开始把 autonomous coding agents 当作软件工程中的新协作者来观察。还有关于开源项目中 AGENTS.md 等 context engineering 文件的研究，说明社区已经意识到：给 agent 提供什么上下文，本身就是一种工程设计。

核心问题：如何让 AI Agent 开发复杂项目？

Anthropic 的实验给了一个很有冲击力的对比：同样是“做一个复古风格的游戏制作工具”，单 Agent 版本大约 20 分钟、花费 9 美元，界面看起来能用，但核心实体之间没有真正连通；完整 harness 跑了大约 6 小时、花费 200 美元，最后得到的是一个真正可运行、能生成 AI 精灵和关卡的应用。

这个对比很重要，因为它说明：AI Agent 的质量差异，不一定体现在第一眼能不能生成页面，而体现在复杂系统里能不能持续发现问题、修复问题、把产品做实。

所以问题不是“AI 会不会写代码”。它当然会。

真正的问题是：如何让 AI Agent 在长期任务中不提前收尾、不自我陶醉、不在错误方向上狂奔，并且能在复杂约束下持续推进？

答案是：不要让 AI Agent 在无边界的空间里自由发挥，而是用约束构建一个有限的、可验证的开发空间。

我把这个方法论称为 “约束多边形”（Constraint Polygon）。

什么是约束多边形？

想象一个多边形。它的每条边代表一种约束：设计文档、UX 原型、测试用例、安全规范、性能基线……这些约束共同围出了一个封闭的空间。

多边形中间是中空的——那就是 AI Agent 需要填充的代码实现。

关键洞察是：当约束足够充分时，代码实现本身反而变成了最确定的部分。AI Agent 不再是在茫茫大海中摸索，而是在一个有明确边界的空间里“自然生长”。

但这里的“约束”不能只是写在需求文档里的愿望清单。它必须具备三种性质：

  1. 可进入上下文：Agent 在执行任务时真的能看到，而不是躺在某个没人读取的文档库里。
  2. 可被外部验证：测试、Playwright、lint、数据库检查、API 调用结果，必须能从外部反驳 Agent 的自我感觉良好。
  3. 可驱动下一轮迭代：评估结果不能只是“好 / 不好”，而要能转化成下一轮明确行动。

这也解释了为什么 OpenAI 会强调“给 Codex 一张地图，而不是一本一千页的说明书”。对于 agent 来说，无法在运行时进入上下文的信息，就近似于不存在。约束必须可访问、可执行、可反馈，才能真正成为驱动力。

逐步构建约束

约束多边形不是一蹴而就的，它是逐步构建的。从最小的三角形开始，根据项目复杂度逐步扩展。

第一步：设计文档——奠定底边

一切从设计开始。在写任何代码之前，先充分理解现有代码仓库的结构和约束，然后产出一份 Dev Design Document。

这份文档要回答：

  • 这个 feature 要实现什么？
  • 它在整个系统中的位置是什么？
  • 它与哪些现有模块交互？
  • 接口契约是什么？
  • 哪些边界绝对不能突破？

这是多边形的底边——最基础的约束。

所有代码都必须有文档。每一段代码在逻辑链条中实现了什么功能、满足了什么设计约束，都需要在文档中说清楚。这不是奢侈品，这是 AI 时代开发的基础设施。

第二步：UX 原型——构建第二条边

在设计文档的基础上，结合 feature 的体验要求，产出 UX 原型。这可以是：

  • HTML/CSS 构建的交互原型
  • 标注了交互逻辑的原型图
  • 明确了用户操作流程的 wireframe
  • 后端功能对应的 API 契约草案

UX 原型和 Dev Design Doc 共同构成了约束的两条边——技术方案约束 + 用户体验约束。

第三步：测试约束——闭合三角形

基于设计文档，定义 测试约束：

  • 在关键节点上，输入什么、输出什么？
  • 调用某个函数时，返回值应该是什么？
  • 点击某个按钮后，界面状态应该如何变化？
  • 什么情况下算成功？什么情况下算失败？

一旦三角形闭合，AI Agent 就有了最小可行的开发空间。 它可以用测试来验证自己的实现，从“测试失败”逐步推进到“测试通过”，直到把三角形填满。

这就是 测试驱动 + AI Agent 的天然结合。AI Agent 最大的问题不是不会写，而是经常“不知道自己错了”。测试给它提供了外部评价器。

不过 Anthropic 的实验也提醒我们：测试约束不应该只理解为单元测试。对于前端和全栈产品，真正有效的 Evaluator 往往要像真实用户一样操作系统：打开页面、点击按钮、检查数据库、调用接口、截图对比。很多 bug 静态看代码并不明显，只有真实跑一遍流程才会暴露。

第四步：扩展更多约束边

当项目更复杂时，三角形不够用。我们可以继续添加约束边：

  • 安全约束：权限模型、数据加密、输入校验、依赖扫描
  • 性能基线：响应时间、吞吐量、资源消耗上限
  • 合规要求：法律法规、行业标准、数据隐私
  • DevOps 约束：CI/CD 流水线规则、部署兼容性、回滚策略
  • 可观测性约束：日志、trace、metrics、告警规则

每一条新的约束边，都让 AI Agent 的实现空间更加精确。

这里还有一个容易被忽略的点：评估语言本身也是约束。 Anthropic 在前端设计实验中发现，只要把“设计质量、原创性、工艺、功能性”这样的评分维度写进生成 Agent 的 prompt，第一轮输出质量就会明显提升；而如果评分标准里出现“博物馆级别设计”这样的描述，生成方向也会随之改变。

也就是说，评价标准不是开发结束后的检查表，而是开发开始前就会塑造结果的方向盘。

POC 桩点：多边形中间的锚定

有时候，约束多边形的外围已经定义清楚了，但中间的填充仍然充满不确定性。技术挑战可能包括：

  • 某个第三方 API 是否真的支持我们的需求？
  • 新的数据库连接方案能否跑通？
  • 多端兼容性是否可行？
  • 浏览器 API 是否覆盖了目标功能？
  • 现有系统的扩展点是否真的能承载新体验？

这时候需要 POC（Proof of Concept）桩点。

关键原则是：不要在巨大的代码仓库中验证技术可行性，而是建立一个最小化的相同架构环境来验证。

POC 桩点的成果是一些小的、可运行的代码模块。它们不是完整的 App，但它们验证了关键技术路径，并作为“锚点”被植入多边形内部，为 AI Agent 的全面填充提供技术确定性。

这其实与 Anthropic 的 planner / generator / evaluator 思路一致：不要让一个 agent 同时承担所有不确定性。规划、验证、生成、评价需要被拆开，分别形成可检查的环节。

完整开发流程

这个流程的本质是：先构建约束，再让 AI 填充。

如果借用 Anthropic 的三 Agent 架构，可以这样理解：

  • Planner 对应约束多边形的初始构建者：把 1–4 句话的模糊需求扩展为产品规格，但避免过早锁死技术细节。
  • Generator 对应多边形内部的填充者：在约束空间中实现代码，并先过滤明显低级错误。
  • Evaluator 对应多边形外部的压力测试者：用真实运行、真实交互、真实检查来判断实现是否成立。

它与传统软件工程的流程高度吻合，但赋予了每个阶段新的意义——不再只是给人类开发者的路标，而是给 AI Agent 的运行边界、评价函数和反馈通道。

为什么这样做有效？

1. 将大问题分解为有界问题

LLM 的上下文窗口有限，但约束多边形把“在百万行代码中实现一个 feature”变成了“在明确约束内填充实现”——后者的上下文可以被精确控制。

这也是缓解 context anxiety 的关键：不要指望一个 Agent 在越来越满的上下文里硬撑到底，而是把任务切成有明确交接物的阶段。阶段之间可以 reset，但设计文档、任务清单、验证结果、未完成事项必须作为结构化 artifact 传递下去。

2. 外部评估让 AI 摆脱自我陶醉

AI Agent 最大的问题是“不知道自己错了”。有了测试约束，AI Agent 可以从失败状态向成功状态逐步推进，这是一个有明确梯度的优化过程。

但更进一步，复杂系统需要的不只是测试，而是独立 Evaluator。它不应该相信 Generator 的自述，而应该通过运行、点击、截图、数据库检查、API 检查来验证结果。生成和评估越分离，系统越不容易陷入 self-evaluation bias。

3. POC 消除了技术不确定性

在正式实现之前，通过轻量级的 POC 验证关键技术路径，避免 AI Agent 在不可行的方向上浪费大量 token 和时间。

4. Harness 把个人能力变成组织能力

如果只是一个高手 prompt 得好，那是个人技巧；如果团队把 design doc、prototype、tests、Playwright evaluator、linters、CI、review bots、observability、rollback 都做成可复用的 harness，那就是组织能力。

Anthropic 文章里还有一句话值得反复咀嚼：harness 里的每个组件，本质上都编码了一个假设——“当前模型自己还做不好什么”。这个假设需要持续压力测试，因为模型变强之后，旧 harness 可能过时；但新的能力边界又会暴露新的 harness 组合空间。

这也是为什么源码泄露不等于飞轮泄露。代码可以被复制，但持续产生高质量代码的组织系统很难被复制。

5. 与现有软件工程实践无缝衔接

这不是一个全新的范式，而是对现有软件工程流程的重新诠释——Design → Prototype → Test → Implement → Review → Observe。每一步都有对应的 AI 时代新意义。

结语：约束即自由

AI Agent 不是来替代软件工程的，而是来重新定义软件工程中“人”和“机器”各自的角色。

人类开发者的新角色：约束的设计者。 我们的工作从“写代码”转变为“定义代码的边界”——设计文档、UX 原型、测试用例、安全规范、性能基线、POC 桩点、评估器、反馈回路……这些约束越精确，AI Agent 的输出就越可靠。

AI Agent 的角色：约束内的填充者。 在人类精心构建的多边形内，AI Agent 可以高效、可验证地完成代码实现。

这就是 AI Agent 时代的开发方法论：不是自由创造，而是约束填充。 不是放手让 AI 自由发挥，而是用工程师的智慧构建精准的约束空间，然后让 AI 在其中自然生长。

Claude Code 泄露给我们的最大启发，不是“原来它的代码是这样写的”，而是“真正的壁垒可能从来就不在单个代码文件里”。

真正的壁垒在于：谁能更快、更稳定、更系统地构建驱动 AI Agent 前进的工程飞轮。

未来的软件工程竞争，很可能不是“谁更会写 prompt”，而是“谁更会设计 harness”：谁能把上下文管理、任务分解、外部评估、POC 验证、持续反馈和工程约束组织成一套可复用系统，谁就能把 AI Agent 从一次性代码生成器，变成真正的长期开发生产力。
