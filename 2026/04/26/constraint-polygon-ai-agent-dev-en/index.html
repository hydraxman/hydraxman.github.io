Constraint Polygon: A Software Engineering Methodology for the AI Agent Era

Claude Code leaked source code, not Anthropic’s engineering flywheel. The real thing worth studying is not “what code did they write,” but “what system keeps the agent moving forward with quality, week after week?”

Preface: Source Code Leaked, but the Flywheel Did Not

A while ago, Claude Code’s source code was accidentally exposed through a source map file in an npm package. Public reports said that a version of @anthropic-ai/claude-code included cli.js.map, allowing outsiders to reconstruct a large amount of readable TypeScript source code, system prompts, context-management logic, and agent-orchestration structure.

This was a meaningful event. For anyone building coding agents, IDE agents, or software engineering automation, it offered a rare chance to look at how a frontier AI company wraps a large model into a real developer tool.

But it did not mean Anthropic’s actual advantage had been copied.

Source code is an output. It is not the engine.

The harder questions were not leaked:

  • How does Anthropic define product requirements?
  • How does it organize agentic engineering?
  • How does Claude Code keep shipping features in weekly increments?
  • How are evaluation, testing, release, rollback, and product feedback connected?
  • How does a team turn model capability into continuous software product momentum?

In other words, what leaked was a diagram of airplane parts, not the engineering system that keeps the airplane accelerating.

That is the software engineering question of the AI era: what should drive software forward when agents can write the code?

For decades, software engineering has proposed many forms of “Driven Development”: TDD, BDD, DDD, FDD, ATDD, MDD, SDD, and more. Each methodology answers the same deeper question:

When system complexity exceeds individual intuition, what should drive development?

In the AI Agent era, this question becomes sharper.

AI can generate code. But it still needs to be driven, constrained, evaluated, and pulled forward by something outside itself. Otherwise, it is merely a fast generator without direction.

From XDD to Harness Engineering: The Driver Is Moving Up the Stack

Different XDD methodologies hand the steering wheel to different artifacts.

Methodology Driver Core artifact Meaning for AI Agents
TDD Tests Unit and integration tests Gives the agent an automatically verifiable target
BDD Behavior Given-When-Then scenarios Turns user behavior into executable semantics
ATDD Acceptance Acceptance criteria Defines business success before implementation
FDD Features Feature list and feature plan Moves delivery forward in feature-sized slices
DDD Domain Ubiquitous language and domain model Reduces semantic drift in complex business systems
MDD Models UML, DSL, or formal models Uses models to generate or constrain implementation
SDD Specification Spec, plan, tasks Makes specification the source of implementation
PDD / Vibe Coding Prompt Natural-language intent Fast, but unstable without enough constraints

GitHub’s Spec Kit documentation captures the shift clearly: when a specification and implementation plan can generate code, development is no longer a loose translation from document to implementation. It becomes a structured conversion from specification to implementation, with templates, checklists, test-first thinking, and constitutional constraints used to keep LLM output maintainable, testable, and architecturally stable.

OpenAI described a similar shift under the term Harness Engineering. In an engineering post, the team described building an internal beta product with Codex over five months under a radical constraint: zero lines of human-written code. The important sentence was not the number itself, but this:

Humans steer. Agents execute.

That is the new division of labor. Humans do not primarily create the system by typing implementation code. Humans create the environment, scaffolding, constraints, documentation, feedback loops, and control systems that make agents productive.

Anthropic’s writing on harness design points in the same direction. Frontier agentic coding performance does not come only from stronger models. It also depends on better harnesses: separating planners, generators, and evaluators; resetting context at the right time; using structured handoffs; and building external evaluation instead of trusting an agent to grade its own work.

Now add one more recent signal: Google Gemini engineering leader Addy Osmani open-sourced Agent Skills, a project that quickly earned tens of thousands of GitHub stars. Its idea is simple but important: package senior engineering workflows into reusable skills so AI coding tools behave less like impatient juniors and more like disciplined engineering teammates.

Agent Skills is not trying to make the model smarter. It is trying to make the surrounding engineering discipline more explicit. Its skills cover the lifecycle from definition and planning to building, testing, review, and shipping. In Claude Code, commands like /spec, /plan, /build, /test, /review, and /ship turn an AI coding session into a structured development pipeline. It also includes specialized agents such as code reviewer, test engineer, and security auditor.

This is exactly the same larger pattern.

  • Spec Kit uses documents to define the agent.
  • Superpowers uses workflow to guide the agent.
  • Agent Skills uses engineering discipline to constrain the agent.
  • Harness Engineering uses the entire operating environment to drive the agent.

So the direction is clear: AI-era software engineering is moving from code-driven development toward harness-driven development.

And a harness is, at its core, a system of constraints that lets an AI Agent work safely, quickly, and verifiably.

The Reality Gap: Demos Are Easy, Mature Products Are Hard

Since 2025, AI coding agents have delivered obvious productivity gains. Cursor, Claude Code, Codex, Copilot, Gemini CLI, and many other tools can create small applications and demos with impressive speed. For projects of dozens, hundreds, or even a few thousand lines, agents often feel almost magical.

The reason is straightforward: when the whole project fits inside the model’s context window, the agent can hold the entire system in mind.

But real software work rarely looks like a demo.

When you work inside a mature codebase with hundreds of thousands or millions of lines, the situation changes completely:

  • Context explodes: the codebase is far larger than the model’s context window.
  • Constraints accumulate: architecture, design systems, security requirements, compatibility promises, and business rules stack on top of one another.
  • Work becomes incremental: you are not starting from a blank page; you are performing surgery on a living system.

It is easy for a brilliant architect to design a building on a blank sheet of paper. It is much harder to add a new floor to an occupied skyscraper without breaking the plumbing, elevators, fire exits, and daily operations.

This is why “agent discipline” projects keep appearing even while models keep improving. The better models become, the more visible their shortcut-taking becomes. Strong agents can move faster in the wrong direction. They need engineering boundaries.

Academic research is converging on the same insight. SWE-agent emphasized the importance of the Agent-Computer Interface: not merely connecting an LLM to a shell, but designing how the agent observes, edits, tests, and receives feedback. Research on AI teammates and Software Engineering 3.0 treats autonomous coding agents as new collaborators in software engineering. Work on AGENTS.md and context engineering shows that the context we provide to agents is itself a form of engineering.

The Core Question: How Do We Let AI Agents Work on Complex Projects?

The answer is: do not let agents roam an unbounded solution space. Build a limited, explicit, verifiable space for them to work inside.

I call this methodology the Constraint Polygon.

What Is a Constraint Polygon?

Imagine a polygon. Each side represents a constraint: design documents, UX prototypes, tests, security policies, performance baselines, compliance rules, observability requirements, and so on. Together, these sides enclose a bounded space.

The center of the polygon is intentionally empty. That is where the AI Agent fills in the implementation.

The key insight is this: when the constraints are rich enough, implementation becomes the most determined part of the process. The agent no longer wanders through an ocean of possible code. It grows code inside a well-defined enclosure.

This also explains why OpenAI emphasizes giving Codex a map rather than a thousand-page manual. For an agent, information that cannot enter runtime context is almost equivalent to information that does not exist. Constraints must be accessible, executable, and feedback-producing before they can truly drive development.

Agent Skills makes the same idea practical. A skill is not just advice. It is a reusable boundary around how work should be done: how to define the task, how to plan it, how to build incrementally, how to test, how to review, and how to decide whether something is ready to ship.

In other words, a skill is a portable side of the polygon.

Building the Polygon Step by Step

A Constraint Polygon is not created all at once. You start with the smallest viable triangle and add more sides as the problem becomes more complex.

Step 1: The Design Document — The Base Side

Everything begins with design. Before writing code, the agent and the human must understand the existing codebase, the feature’s purpose, and the boundaries that must not be crossed. This produces a Dev Design Document.

The design document should answer:

  • What is this feature supposed to do?
  • Where does it fit in the system?
  • Which existing modules does it interact with?
  • What are the interface contracts?
  • Which boundaries are non-negotiable?

This is the base side of the polygon.

In the AI Agent era, documentation is not ceremony. It is runtime infrastructure. Every important piece of code should be traceable to the design constraint it satisfies. If the agent cannot see the design, the design does not guide the agent.

This is also where Agent Skills and Spec Kit become useful. /spec and /plan are not just productivity shortcuts. They create the artifacts that later constrain implementation.

Step 2: The UX or API Prototype — The Second Side

On top of the design document, create an explicit prototype of the user experience or API behavior. This could be:

  • an HTML/CSS interaction prototype,
  • a wireframe with annotated interaction logic,
  • a user-flow diagram,
  • an API contract draft,
  • a CLI command transcript,
  • or an example request/response pair.

The design document and prototype together form two sides of the polygon: technical intent and user-facing behavior.

This matters because agents often optimize for local instruction completion. If the user experience is vague, they may satisfy the literal prompt while missing the product feel. A prototype makes the expected experience visible.

Step 3: Test Constraints — Closing the Triangle

Based on the design and prototype, define tests:

  • Given an input, what output should a function return?
  • After a button click, what UI state should exist?
  • What is success? What is failure?
  • What edge cases must not regress?
  • What security-sensitive behavior must be rejected?

Once tests exist, the triangle closes. The agent now has a minimum viable development space. It can move from failing tests to passing tests, using test results as an external evaluator.

This is where TDD and AI Agents fit together naturally. The agent’s biggest weakness is not that it cannot write. It is that it often does not know when it is wrong. Tests give it an external gradient.

Agent Skills captures this discipline by making /test and specialized test-engineer agents part of the workflow. The point is not merely to run tests after the fact. The point is to make testability part of the driving system.

Step 4: Add More Constraint Sides

For more complex systems, a triangle is not enough. Add more sides:

  • Security constraints: permission models, input validation, dependency scanning, data encryption.
  • Performance baselines: response latency, throughput, memory limits, bundle size.
  • Compliance requirements: privacy rules, legal constraints, industry standards.
  • DevOps constraints: CI/CD rules, deployment compatibility, rollback strategy.
  • Observability constraints: logging, tracing, metrics, alerts.
  • Code-review constraints: naming rules, architectural boundaries, dependency rules.
  • Release constraints: changelog, migration plan, rollout guardrails.

Each new side shrinks the agent’s solution space and makes the implementation more precise.

This is the deeper significance of Agent Skills: it decomposes senior engineering judgment into reusable operational constraints. Code reviewer, test engineer, and security auditor are not just personas. They are additional evaluator sides around the implementation space.

POC Anchors: Fixing Uncertainty Inside the Polygon

Sometimes the outer boundary is clear, but the center is still uncertain. The risky questions might be:

  • Does a third-party API really support this use case?
  • Can the new database connection strategy work under our constraints?
  • Does the browser API cover the required behavior?
  • Can the current extension point support the new experience?
  • Will a performance-sensitive path remain fast enough?

This is where POC anchors matter.

The principle is simple: do not validate technical feasibility inside a huge production codebase. Build a small, controlled environment with the same architectural conditions and test the risky path there.

A POC anchor is a small runnable module. It is not the full app, but it proves a critical path and becomes an anchor point inside the polygon. Once the anchor exists, the agent can fill around it with far less uncertainty.

This aligns with Anthropic’s planner / generator / evaluator separation. Do not force one agent to carry planning, uncertainty elimination, implementation, and evaluation at the same time. Split them into checkable stages.

The Full Development Flow

The flow is simple:

  1. Understand the codebase.
  2. Write the design document.
  3. Create the UX/API prototype.
  4. Define tests and acceptance criteria.
  5. Add security, performance, compliance, DevOps, and observability constraints.
  6. Build POC anchors for uncertain technical paths.
  7. Let the AI Agent implement inside the polygon.
  8. Run tests, review, security audit, and release checks.
  9. Feed production feedback back into the harness.

This is not a rejection of traditional software engineering. It is a reinterpretation of it for the agent era.

Design, prototype, test, implement, review, ship, observe—these steps already existed. What changes is that they no longer only guide human developers. They become the operating boundary, evaluation function, and feedback channel for AI Agents.

Why This Works

1. It Turns an Unbounded Problem into a Bounded One

A large codebase may exceed any context window. But the Constraint Polygon converts “implement a feature somewhere in a million lines of code” into “fill this bounded implementation space under explicit constraints.” That bounded space can be represented compactly.

2. Tests Let Agents Self-Improve Safely

Agents need external feedback. Tests give them a clear path from failure to success. Without tests, they may produce confident but wrong work. With tests, they can iterate.

3. POC Anchors Remove Technical Uncertainty

Many failures happen because an agent assumes feasibility that has not been proven. POC anchors turn unknowns into knowns before the full implementation begins.

4. Harnesses Turn Individual Skill into Organizational Capability

A clever prompt is a personal trick. A repeatable system of design docs, prototypes, tests, linters, CI, review agents, security audits, observability, and rollback policies is organizational capability.

This is why leaked source code does not equal a leaked flywheel. Code can be copied. A system that continuously produces high-quality code is much harder to copy.

5. It Explains Why Agent Skills Matter

Agent Skills, Spec Kit, Superpowers, and similar projects keep appearing because teams are discovering the same truth: model intelligence is necessary, but not sufficient.

Production software requires discipline.

Agent Skills packages discipline into reusable skills. Spec Kit packages intent into structured specifications. Superpowers packages development into a workflow. Harness Engineering packages the entire environment into a control system.

The Constraint Polygon is a way to understand all of them as parts of the same framework.

Practical Example: A Feature Request in a Mature Product

Suppose you ask an AI Agent to add “team-level billing alerts” to a mature SaaS product.

A weak prompt says:

Add billing alerts for teams.

A Constraint Polygon approach creates:

  • a design doc explaining team billing ownership, existing billing modules, and notification boundaries;
  • an API contract for creating, updating, and testing alert thresholds;
  • a UX prototype showing how admins configure alerts;
  • unit tests for threshold logic;
  • integration tests for notification delivery;
  • security constraints around team-admin permissions;
  • performance constraints around billing-event ingestion;
  • observability requirements for alert delivery failures;
  • a POC anchor validating the notification provider’s rate limits.

Only then does the agent implement.

The agent is still writing code. But it is no longer inventing the world. It is filling a defined space.

A Better Mental Model: Constraint Is Freedom

It sounds paradoxical, but constraints make agents more useful.

Without constraints, the agent has too many degrees of freedom. It can satisfy the prompt while damaging architecture, skipping edge cases, or creating future maintenance debt.

With constraints, the agent can move faster because it no longer has to guess the shape of success.

This is also how senior engineers work. They do not merely type better code. They know which constraints matter before code exists. They know what must be decided, what must be tested, what can be deferred, and what must never be broken.

Agent Skills tries to capture that senior-engineer discipline. The Constraint Polygon explains where that discipline fits: every skill adds another side, anchor, or evaluator to the agent’s working space.

Conclusion: Humans Design Constraints, Agents Fill Them

AI Agents are not replacing software engineering. They are forcing us to make software engineering more explicit.

The human developer’s role is shifting from “writer of code” to designer of constraints. We define design documents, UX prototypes, test cases, security rules, performance baselines, POC anchors, evaluator agents, CI gates, and feedback loops.

The AI Agent’s role is becoming the filler inside the constraint space. Inside a well-designed polygon, it can produce implementation quickly and verifiably.

That is the development methodology of the AI Agent era:

Do not let AI freely improvise. Build the constraint polygon first, then let AI fill it.

Claude Code’s leaked source code taught us one thing: the moat is probably not in any single file. The moat is in the flywheel that drives agents forward.

Agent Skills, Spec Kit, Superpowers, Harness Engineering, and the Constraint Polygon all point to the same future:

The next generation of software engineering will be won by teams that can design better constraints, better context, and better feedback loops for agents.
